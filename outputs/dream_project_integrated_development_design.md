# 29 WITH - 꿈프 통합 개발 설계서

## 1. 문서 목적

이 문서는 지금까지 정리한 꿈나무양성프로젝트 기능을 실제 개발 가능한 구조로 다시 설계한 통합 문서다.

범위:

- 선생님 페이지
- 관리자 페이지
- 백엔드 데이터 구조
- 자동화/메일/외부 연동
- 프론트엔드 화면 구조
- UIUX 원칙
- 예외 처리와 팝업
- 개발 우선순위

핵심 목표:

- 관리자는 여러 행사를 빠르게 운영하고 처리할 수 있어야 한다.
- 선생님은 내 학교의 신청, 선정, 출품, 혜택, 문서 상태를 불안 없이 확인할 수 있어야 한다.
- 29초영화제/29역숏폼왕이 여러 개 동시에 진행되어도 출품작이 섞이지 않아야 한다.
- 여러 꿈프가 동시에 진행되어도 관리자와 선생님이 현재 보고 있는 행사 기준을 헷갈리지 않아야 한다.
- 모든 자동 처리 결과는 이력으로 남아야 한다.

---

## 2. 제품 구조

서비스명:

```text
29 WITH
```

역할:

- 29 Platform 대외협력 통합 관리 시스템
- 현재 1차 범위는 꿈나무양성프로젝트
- 추후 서포터즈, 명예홍보대사, 파트너사까지 확장 가능

상위 카테고리:

```text
전체
꿈프
서포터즈
명예홍보대사
파트너사
```

1차 개발 범위:

```text
꿈프
```

확장 원칙:

- 꿈프 전용 기능은 `dream_project` 도메인으로 분리한다.
- 29 WITH 공통 기능은 `users`, `events`, `notifications`, `files`, `audit_logs`처럼 공통 도메인으로 설계한다.
- 서포터즈/명예홍보대사/파트너사가 추가되어도 사용자, 메일, 파일, 이력 구조는 재사용할 수 있어야 한다.

## 3. 사용자 역할과 권한

### 3.1 관리자

역할:

- 29초영화제 사무국
- 꿈프 운영 관리자

관리자 권한 구분:

- 최고 관리자
- 꿈프 운영 관리자
- 메일/공지 관리자
- 혜택/쿠폰 관리자
- 조회 전용 관리자

권한 원칙:

- 위험 버튼은 권한이 있는 관리자에게만 보인다.
- 상태 변경, 수동 매칭, PDF 재발급, 종료 처리는 작업 이력을 남긴다.
- 조회 전용 관리자는 상태 변경 버튼을 볼 수 없다.

1차 운영 권한 정책:

- 등록된 모든 관리자는 선정 처리, 수동 매칭, 활동확인서 승인, 꿈프 종료를 할 수 있다.
- 메인 관리자는 관리자를 추가하거나 삭제할 수 있다.
- 모든 관리자 작업은 작업 이력에 남긴다.
- 초기 메인 관리자 계정은 별도 보안 절차로 생성한다.
- 관리자 비밀번호는 평문 저장 금지이며 반드시 해시로 저장한다.
- 초기 비밀번호는 최초 로그인 후 변경하도록 안내한다.

### 3.2 선생님

역할:

- 꿈나무양성프로젝트 담당자
- 학교 선생님 또는 단체 담당자

선생님 권한:

- 신청 가능한 행사 조회
- 꿈프 신청
- 내 신청/선정 상태 확인
- 내 학교 출품 상태 확인
- 작품 링크 직접 입력
- 문의/요청 등록
- 구독권/간식비/활동확인서/심사표 확인
- 종료된 프로젝트 문서 다운로드

선생님 화면 제한:

- 다른 학교 정보 노출 금지
- 관리자 메모 노출 금지
- 외부 연동 오류 상세 노출 금지
- 원본 심사 점수 노출 금지

## 4. 전체 운영 흐름

```text
관리자 행사 생성
→ 외부 연동/메일/템플릿 준비
→ 꿈프 모집 시작
→ 선생님 신청
→ 관리자 신청 검토
→ 선정/예비/미선정 처리
→ 선정 학교 담당자 페이지 생성
→ 출품 마감 D-day 안내
→ 외부 출품작 자동 동기화
→ 학교별 출품작 자동/수동 매칭
→ 출품 마감 후 최종 리스트 승인
→ 활동확인서 PDF 생성/발송
→ 수상작 발표 후 심사 결과 반영
→ 심사표 PDF 생성/발송
→ 간식비 지급 상태 입력
→ 종료 전 점검
→ 꿈프 종료
→ 히스토리 저장 및 학교별 참여 이력 반영
```

가장 중요한 연결:

1. 모집 시작 → 신청 버튼 활성화 + 모집 안내 메일
2. 신청 완료 → 선생님 대시보드 프로젝트 카드 생성
3. 선정 처리 → 메일 + 쿠폰 + 담당자 페이지 + 작품 슬롯
4. 출품 동기화 → 행사 식별값 + 소속명/팀명 완전 일치 매칭
5. 최종 승인 → 활동확인서 발급 가능
6. 심사 결과 반영 → 순위/본심 + 심사표 발급
7. 꿈프 종료 → 히스토리 + 참여 이력 반영

## 4.0 운영 정책 확정 사항

### 4.0.1 같은 학교의 여러 꿈프 동시 신청

허용한다.

정책:

- 같은 학교가 여러 꿈프에 동시에 신청할 수 있다.
- 각 신청은 `dream_event_id`와 `participation_id` 기준으로 분리한다.
- 선생님 화면에서는 참여 중인 꿈프를 카드로 분리해서 보여준다.
- 관리자 화면에서는 먼저 꿈프 행사를 선택한 뒤 해당 행사 기준으로 관리한다.

### 4.0.2 같은 학교의 담당 선생님 2명 이상 신청

자동 차단하지 않는다.

운영 방식:

- 관리자가 중복 신청 가능성을 확인한다.
- 사무국에서 별도 연락해 1명이 신청하도록 유도한다.
- 담당 선생님이 실제로 2명인 경우, 출품 시 소속명/팀명에 A, B 등 구분값을 붙이도록 안내한다.

예시:

```text
방학중학교 A
방학중학교 B
```

개발 반영:

- 같은 학교명, 같은 연락처, 유사한 출품 소속명/팀명이 있으면 관리자에게 `중복 가능성`으로 표시한다.
- 자동 병합하지 않는다.
- 관리자 확인 후 선정 또는 정리한다.

### 4.0.3 선정 기준

기본은 선착순이다.

단, 과거 참여 이력에 따라 순위가 조정될 수 있다.

정책:

- 성실 참여 학교/선생님은 선착순 선정에서 우선 검토될 수 있다.
- 미이행 이력이 있는 학교/선생님은 선착순 순위에서 뒤로 밀릴 수 있다.
- 활동 우수 학교 선정 시 성실 참여 이력은 긍정적으로 반영한다.

관리자 화면 표시:

- 신청 순번
- 참여 신뢰 상태
- 선정 추천 순위
- 순위 조정 사유

선생님 화면 표시:

- `성실 참여`
- `우선 검토 대상`
- `확인 필요 이력 있음`
- `우선순위 조정 가능`

선생님 화면에서는 `패널티`라는 표현을 직접 크게 노출하지 않는다.

### 4.0.4 학교명/출품 소속명 변경

신청 전에는 재확인을 통해 중요성을 강조한다.

신청 후 변경은 관리자와 선생님이 함께 이중 확인해야 한다.

정책:

- 신청서 제출 전 학교명과 출품 소속명/팀명을 최종 확인 팝업으로 다시 보여준다.
- 신청 후 학교명 또는 출품 소속명/팀명 변경은 선생님 요청으로 시작한다.
- 관리자는 변경 요청을 확인하고, 기존 출품 매칭 영향도를 확인한다.
- 변경 후 선생님 화면에 변경된 기준명을 다시 확인하도록 표시한다.
- 변경 전/후 값과 확인 이력을 모두 저장한다.

중요 안내 문구:

```text
출품 소속명/팀명은 실제 출품 시 입력하는 이름과 띄어쓰기까지 동일해야 합니다.
신청 후 변경이 필요한 경우 관리자 확인과 담당 선생님 재확인이 필요합니다.
```

## 4.1 동시 진행 꿈프 UX 원칙

꿈프는 동시에 여러 개 진행될 수 있다.

예시:

```text
제13회 박카스 29초영화제 꿈프
제1회 동반성장 29역숏폼왕 꿈프
간단요리사 29역숏폼왕 꿈프
```

같은 학교가 여러 꿈프에 동시에 참여할 수도 있다.

따라서 모든 화면은 아래 원칙을 지켜야 한다.

### 4.1.1 관리자 화면 원칙

관리자 화면은 두 단계로 나눈다.

```text
전체 꿈프 보기
→ 특정 꿈프 선택
→ 선택한 꿈프 기준으로 관리
```

관리자 화면에서 항상 보여야 하는 것:

- 현재 선택한 꿈프 행사명
- 행사 유형
- 운영 상태
- 출품 마감일
- 다른 꿈프로 전환하는 선택 영역

상단 고정 표시 예시:

```text
제13회 박카스 29초영화제
29초영화제 · 모집중 · 출품 마감 D-5
```

행사 미선택 상태:

- 꿈프 대시보드만 보여준다.
- 신청/선정, 출품 확인, 활동확인서, 심사 결과 같은 세부 메뉴는 행사 선택 전에는 비활성화하거나 안내 화면을 보여준다.

행사 선택 후:

- 좌측 메뉴의 모든 화면은 선택한 행사 기준으로만 데이터를 보여준다.
- 다른 행사 데이터는 섞어서 보여주지 않는다.
- 화면 제목에 행사명을 항상 포함한다.

피해야 할 UX:

- 한 화면에 여러 행사의 신청자 리스트를 섞어서 보여주기
- 출품 확인 화면에서 행사 선택 없이 학교명 검색만 제공하기
- 활동확인서/심사표 화면에서 현재 행사명이 작게만 보이기
- 선택한 행사와 다른 행사 데이터가 같은 테이블에 섞이기

### 4.1.2 선생님 화면 원칙

선생님은 여러 꿈프에 동시에 참여할 수 있다.

선생님 화면은 아래 구조를 따른다.

```text
내 대시보드
→ 진행 중인 참여 프로젝트 카드
→ 특정 프로젝트 선택
→ 해당 프로젝트의 작품/혜택/문서 확인
```

선생님 대시보드에는 참여 중인 꿈프를 카드로 분리한다.

카드 표시 정보:

- 행사명
- 행사 유형
- 신청/선정 상태
- 출품 마감 D-day
- 출품 확인 수
- 미확인 수
- 다음 할 일

카드 예시:

```text
제13회 박카스 29초영화제
선정 완료 · 출품 마감 D-5
신청 5편 · 확인 4편 · 미확인 1편
[작품 상태 보기]
```

```text
제1회 동반성장 29역숏폼왕
선정 완료 · 출품 마감 D-12
신청 3편 · 확인 0편 · 미확인 3편
[작품 상태 보기]
```

프로젝트 선택 후:

- 학생/작품 관리, 혜택/지원, 활동확인서, 심사표는 선택한 프로젝트 기준으로만 보여준다.
- 화면 상단에 행사명과 유형을 항상 표시한다.
- 다른 프로젝트로 이동하는 버튼을 제공한다.

피해야 할 UX:

- 여러 행사 작품 슬롯을 한 화면에 섞어서 보여주기
- 간식비, 구독권, 확인서 상태를 전체 합산으로만 보여주기
- 선생님이 어떤 행사 기준의 활동확인서를 보고 있는지 알 수 없게 만들기

### 4.1.3 공통 데이터 원칙

프론트엔드와 백엔드 모두 아래 기준을 지킨다.

```text
모든 신청 데이터는 dream_event_id 기준
모든 참여 데이터는 participation_id 기준
모든 외부 출품 데이터는 external_event_id 기준
모든 출품 매칭은 dream_event_id + external_event_id + participation_id 기준
```

절대 하면 안 되는 것:

```text
학교명만으로 출품작 조회
학교명만으로 확인서 생성
학교명만으로 심사표 생성
학교명만으로 간식비 지급 상태 조회
```

학교명은 표시와 보조 검색에는 사용할 수 있지만, 최종 연결 기준이 되어서는 안 된다.

## 5. 백엔드 핵심 도메인

### 5.1 사용자

주요 테이블:

```text
users
teacher_profiles
admin_users
admin_roles
```

`users`:

- id
- user_type
- name
- email
- phone
- password_hash
- status
- created_at
- updated_at

`teacher_profiles`:

- id
- user_id
- school_name
- school_level
- teacher_name
- subject
- contact_phone
- contact_email
- affiliation_name
- bank_copy_file_id
- id_card_file_id
- affiliation_change_locked
- created_at
- updated_at

정책:

- 회원가입 시 학교 정보는 받지 않는다.
- 학교 정보는 첫 신청 시 프로필로 입력한다.
- 학교명과 출품 소속명/팀명은 출품 확인 기준이므로 신청 후 직접 수정 불가.
- 변경은 관리자 승인 요청으로 처리한다.

### 5.2 행사

- URL도 없거나 변경될 수 있으면 `external_event_id + affiliation_name + title + participant_name + submitted_at` 조합을 보조 키로 사용한다.
- 동기화 시 새 작품, 수정된 작품, 삭제/숨김 가능 작품을 구분한다.
- 기존에 매칭된 작품은 외부 데이터가 일부 변경되어도 바로 삭제하지 않고 `확인 필요`로 표시한다.
- 작품명, 감독/출품자명, URL, 점수, 수상결과가 변경되면 변경 이력을 남긴다.
- 동기화 결과가 이전 대비 급격히 감소하거나 증가하면 자동 반영하지 않고 관리자 검토 대기 상태로 둔다.

- 발송 완료된 메일은 원본 수정 불가.
- 수정본 재발송으로 처리한다.
- 발송 이력, 실패 이력, 재발송 이력을 저장한다.
- D-day 메일은 `행사 + 학교 + 메일유형` 기준으로 중복 발송을 방지한다.

발신자 관리:

- 관리자 개인 관리 페이지에서 발신자 이름과 발신자 이메일을 관리할 수 있다.
- 행사별 기본 발신자 주소를 설정할 수 있다.
- 발신자 주소 변경 이력을 저장한다.
- 실제 메일 발송 도메인 인증 설정은 별도 시스템 설정에서 관리한다.

수신자 정책:

- 선생님 회원가입 시 업무용 이메일 사용을 권장하는 안내를 표시한다.
- 기본 수신자는 회원 이메일이다.
- 행사별 추가 수신자를 등록할 수 있다.
- 같은 내용의 메일은 여러 선생님에게 한 번에 발송할 수 있다.
- 대량 발송 전 대상 학교 수, 수신자 수, 제외 대상 수를 확인한다.

메일 문구 관리:

- 단계별 메일 문구 초안을 템플릿으로 저장한다.
- 관리자는 템플릿을 추가, 삭제, 수정할 수 있다.
- 템플릿 수정 이력을 저장한다.
- 예약 메일은 발송 전 제목, 본문, 발송일, 발송 시간, 발신자, 수신자를 확인하고 수정할 수 있다.

- 확인서에 들어가는 텍스트는 관리자 수정 가능해야 한다.
- 수정 가능한 항목은 학교명, 작품명, 학생명, 활동 기간, 발급일, 제호, 본문 문구다.

## 6. 관리자 프론트엔드 설계

### 6.1 전체 구조

관리자 상단 카테고리:

```text
전체 | 꿈프 | 서포터즈 | 명예홍보대사 | 파트너사
```

꿈프 선택 후 좌측 메뉴:

```text
꿈프 대시보드
행사 운영
회원/참여 학교
신청/선정
출품 확인
혜택/지원
활동확인서
심사 결과
메일/공지
문의/요청
연동 설정
히스토리
```

좌측 하단:

```text
진행 중인 꿈프 행사 목록
```

행사를 선택하면 모든 메뉴의 기준이 해당 행사로 변경된다.

### 6.2 관리자 꿈프 대시보드

목적:

- 오늘 처리해야 할 행사와 작업을 빠르게 확인

표시:

- 진행 중인 꿈프 카드
- 처리 필요 건수
- 출품 확인 진행률
- 마감 임박 상태
- 메일 실패/재발송 필요
- 간식비 예정일 미입력

주요 버튼:

- `출품 확인`
- `처리 필요 보기`
- `행사 보기`

UIUX:

- 처리 필요 건이 있는 행사를 먼저 정렬한다.
- 다른 프로젝트 카테고리의 상세 정보는 보여주지 않는다.
- 대시보드는 보고서가 아니라 작업 출발점이다.

### 6.3 행사 운영

목적:

- 선택한 행사 하나의 운영 단계 확인

표시:

- 행사명
- 행사 유형
- 공모기간
- 출품 마감일
- 꿈프 운영 상태
- 운영 단계 체크리스트

단계별 주요 버튼:

- 모집 전: `꿈프 모집 시작`
- 신청 기간: `신청자 보기`
- 선정 전: `선정 처리`
- 출품 중: `출품 확인`
- 마감 후: `최종 리스트 승인`
- 승인 후: `활동확인서 PDF 생성`
- 심사 후: `심사표 생성`
- 모든 완료 후: `꿈프 종료`

UIUX:

- 현재 단계에서 필요한 버튼 1개를 가장 크게 보여준다.
- 완료된 단계는 접어둔다.

### 6.4 신청/선정

목적:

- 신청 학교 검토와 선정 처리

구성:

- 상단 요약
- 필터
- 학교 단위 리스트
- 우측 상세 패널

리스트:

- 신청일
- 학교명
- 담당교사
- 출품 소속명/팀명
- 예상 출품편수
- 참여 상태
- 신청 상태
- 다음 행동

주요 버튼:

- `선정 처리`
- `예비 처리`
- `미선정 처리`
- `메일 미리보기`
- `엑셀 다운로드`

중요 동작:

- 선정 처리 시 메일, 쿠폰, 담당자 페이지, 작품 슬롯이 함께 생성된다.
- 쿠폰 부족 시 선정은 가능하되 구독권은 지급 예정으로 둔다.

### 6.5 출품 확인

목적:

- 학교별 출품 확인 상태와 미확인 작품 조치

구성:

- 학교 단위 요약 리스트
- 학교 선택 시 작품 상세

리스트:

- 학교명
- 신청 작품 수
- 확인 수
- 미확인 수
- 초과 출품 수
- 매칭 상태
- 마지막 동기화 시간

주요 버튼:

- `전체 동기화`
- `선택 학교 재확인`
- `수동 매칭`
- `선생님에게 확인 요청`

UIUX:

- 첫 화면에서 전체 작품 목록을 펼치지 않는다.
- 학교당 작품이 많으므로 학교 단위 요약을 먼저 보여준다.
- 유사 이름은 후보로만 보여주고 자동 매칭하지 않는다.

### 6.6 혜택/지원

목적:

주요 버튼:

- `간식비 지급 예정일 입력`
- `지급 완료 처리`
- `지원 안내 메일 발송`

UIUX:

- 미입력 학교 필터가 필요하다.
- 내부 사유와 선생님 노출 문구를 분리한다.

### 6.7 활동확인서

- `활동확인서 PDF 생성`
- `발급 메일 발송`

UIUX:

- 승인 전에는 다운로드 버튼 없음.
- 학생 이름 누락 시 PDF 생성 비활성.

## 7. 선생님 프론트엔드 설계

### 7.1 전체 메뉴

```text
내 대시보드
신청 가능한 행사
참여 프로젝트
학생/작품 관리
혜택/지원
공지/메일함
활동확인서
심사표
참여 이력
내 프로필
```

원칙:

- 내 학교 기준으로만 보여준다.
- 첫 화면은 불안 해소가 목표다.
- 문제가 없으면 “정상 진행 중”을 명확히 보여준다.
- 문제가 있으면 다음 행동 하나만 강조한다.

### 7.2 내 대시보드

상단:

- 참여 행사명
- 신청 접수 상태
- 선정 상태
- 참여 신뢰 상태
- 출품 마감 D-day

핵심 카드:

- 신청 작품 수
- 출품 확인 수
- 미확인 수
- 다음 할 일
- 구독권 상태
- 간식비 지급 예정일
- 활동확인서 상태
- 심사표 상태

주요 버튼:

- 미확인 작품 있음: `작품 상태 보기`
- 선정 전: `신청서 보기`
- 문서 발급 후: `PDF 다운로드`

숨김:

- 관리자 내부 상태
- 외부 연동 오류 상세
- 다른 학교 정보

### 7.3 신청 가능한 행사

리스트 표시:

- 행사명
- 행사 유형
- 공모기간
- 상태

버튼:

- `자세히 보기`
- `신청하기`
- `신청 대기`
- `신청 완료`

정책:

- 모집 전에는 신청 버튼 비활성.
- 이미 신청한 행사는 중복 신청 불가.

### 7.4 행사 상세/신청

상세:

- 포스터
- 행사명
- 행사 유형
- 공모기간
- 주제
- 안내사항
- 홈페이지 버튼
- 신청하기

신청서:

- 프로필 정보
- 출품 소속명/팀명
- 예상 출품편수
- 활용 계획
- 출품 예정일
- 메모

중요 안내:

```text
출품 소속명/팀명은 실제 출품 시 입력하는 이름과 띄어쓰기까지 동일해야 합니다.
```

제출 후:

- 신청 완료 화면 표시
- 신청 완료 메일 발송
- 대시보드 프로젝트 카드 생성

### 7.5 학생/작품 관리

표시:

- 출품 소속명/팀명 + 복사 버튼
- 신청 작품 수
- 확인 작품 수
- 미확인 작품 수
- 작품 슬롯

작품 상태:

- 출품전
- 확인대기
- 확인완료
- 직접입력필요
- 관리자확인중

버튼:

- `작품 링크 입력`
- `관리자에게 확인 요청`
- `소속명/팀명 안내`

UIUX:

- 확인 완료 작품은 작게 접는다.
- 미확인 작품만 크게 보여준다.

### 7.6 혜택/지원

표시:

- 구독권 쿠폰번호
- 구독권 지급 상태
- 간식비 지급 상태
- 간식비 지급 예정일
- 금액
- 활동확인서 상태
- 심사표 상태

UIUX:

- 간식비는 “언제 받을 수 있는지”를 가장 먼저 보여준다.
- 보류/미대상은 짧은 이유와 문의 버튼만 제공한다.

### 7.7 공지/메일함

표시:

- 메일 제목
- 발송일
- 관련 행사
- 읽음 여부

버튼:

- `보기`
- `관련 화면으로 이동`

정책:

- 이메일을 놓쳐도 같은 내용을 페이지에서 확인 가능.
- 메일 실패 로그는 선생님에게 노출하지 않는다.

### 7.8 활동확인서

상태:

- 발급 전
- 관리자 확인 중
- 다운로드 가능
- 재발급 준비 중

버튼:

- `PDF 다운로드`
- `수정 요청`
- `재발급 요청`

정책:

- 승인 전에는 다운로드 버튼을 숨긴다.
- 왜 다운로드할 수 없는지 짧게 안내한다.

### 7.9 심사표

상태:

- 발표 전
- 결과 반영 완료
- 다운로드 가능

표시:

- 작품명
- 순위
- 본심 여부

정책:

- 29역숏폼왕은 본심 항목 숨김.
- 원본 점수 숨김.

### 7.10 참여 이력

표시:

- 현재 참여 상태
- 신청 횟수
- 선정 횟수
- 출품 완료 횟수
- 미이행 횟수
- 최근 참여 프로젝트

표현:

- 성실 참여
- 우선 검토 대상
- 확인 필요 이력 있음
- 우선순위 조정 가능

정책:

- `패널티` 표현은 직접 노출하지 않는다.

## 9. 팝업/예외 처리

### 9.1 선생님 팝업

반드시 팝업:

- 신청서 최종 제출
- 학교명 변경 요청
- 출품 소속명/팀명 변경 요청
- 학교명과 출품 소속명/팀명이 다를 때 신청 제출
- 활동확인서 수정/재발급 요청
- 마감 후 작품 링크 제출

조건부 팝업:

- D-5 이하 미확인 작품 있음
- 마감 당일
- 간식비 지급 보류 최초 안내
- 예상 출품편수가 비정상적으로 큼
- 모집 목표 작품 수 초과

팝업 없이 화면 안내:

- 신청 검토 중
- 구독권 지급 준비 중
- 간식비 지급 일정 확인 중
- 심사표 발표 전
- 활동확인서 승인 전
- 외부 연동 관리자 확인 중

### 9.2 관리자 팝업

반드시 팝업:

- 꿈프 모집 시작
- 모집 중 행사 주요 정보 수정
- 선정/예비/미선정 처리
- 선정 상태 되돌리기
- 대량 메일 발송
- 예약 메일 즉시 발송
- 수동 매칭
- 이미 연결된 작품의 연결 변경
- 최종 승인
- 활동확인서 재발급
- 심사 결과 반영
- 쿠폰 중복 지급
- 간식비 일괄 변경
- 꿈프 종료
- 예외 종료

조건부 팝업:

- 외부 연동 미완료 상태에서 모집 시작
- 쿠폰 부족 상태에서 선정
- 중복 신청 가능성 있음
- 동기화 결과 급변
- 심사 결과 급변
- 출품 마감 전 예외 승인

### 9.3 일부 성공/일부 실패 처리

예시:

```text
선정 상태 변경 성공
담당자 페이지 생성 성공
메일 발송 실패
쿠폰 지급 실패
```

정책:

- 성공한 작업은 유지한다.
- 실패한 작업은 관리자 작업함에 생성한다.
- 결과 요약 팝업을 보여준다.
- 선생님 화면에는 내부 실패를 직접 노출하지 않는다.

## 10. 자동화와 배치

### 10.1 예약 메일

- 테스트 발송 가능
- 즉시 발송 가능
- 예약 취소 가능
- 발송 이력 저장

### 10.2 출품 동기화

동기화 방식:

- 행사별 외부 연동 설정 기준
- 서버 접근 정보는 안전하게 저장
- 외부 관리자 페이지 URL과 외부 행사 식별값은 행사별로 저장

검증:

- 외부 행사 식별값 불일치 시 중단
- 필수 필드 누락 시 중단
- 동기화 결과 급변 시 검토 대기

### 10.3 종료 처리

종료 전 체크:

- 신청/선정 상태 확정
- 최종 출품 리스트 승인
- 활동확인서 발급 완료 또는 미대상 처리
- 심사표 발급 완료 또는 미대상 처리
- 구독권 지급 상태 확인
- 간식비 지급 상태 입력
- 미이행 학교 반영
- 문의/요청 미처리 없음
- 메일 발송 실패 건 처리

종료 시:

- 꿈프 상태 종료
- 진행 중 목록 제거
- 히스토리 저장
- 학교별 참여 이력 업데이트
- 선생님 종료 프로젝트로 이동
- 문서 다운로드 유지

## 12. 개발 시 반드시 지켜야 할 핵심 규칙

1. 학교명만으로 출품작을 매칭하지 않는다.
2. 자동 매칭은 완전 일치만 허용한다.
3. 행사 식별값 없이 외부 동기화를 실행하지 않는다.
4. 선정 처리는 상태 변경만 하지 않고 메일, 쿠폰, 담당자 페이지, 작품 슬롯을 함께 처리한다.
5. 자동 메일은 발송 전 확인/수정 가능한 예약 메일로 관리한다.
6. 활동확인서 다운로드는 관리자 최종 승인 후에만 가능하다.
7. 학생 이름 누락 시 활동확인서 PDF를 생성하지 않는다.
8. 29역숏폼왕에는 본심 여부를 표시하지 않는다.
9. 꿈프 종료 시 히스토리와 학교별 참여 이력을 반드시 업데이트한다.
10. 관리자 수동 조치와 상태 변경은 모두 작업 이력을 남긴다.

## 13. 최종 설계 판단

29 WITH 꿈프 기능은 단순 신청 관리가 아니라 아래 네 가지가 결합된 운영 시스템이다.

```text
신청/선정 관리
외부 출품작 연동
문서/혜택 자동 발급
학교별 참여 이력 관리
```

따라서 프론트엔드는 사용자별로 필요한 정보만 보여주고, 백엔드는 상태 전환과 자동 처리 이력을 안정적으로 관리해야 한다.

관리자 화면의 핵심:

```text
오늘 처리할 일
다음에 누를 버튼
오류 조치
작업 이력
```

선생님 화면의 핵심:

```text
내 신청 상태
내 선정 상태
내 출품 확인 상태
내 혜택/문서 상태
내가 지금 해야 할 일
```

이 기준으로 개발하면 기능이 많아져도 화면이 복잡해지지 않고, 운영 중 오류가 발생해도 어디서 조치해야 하는지 명확하게 유지할 수 있다.
