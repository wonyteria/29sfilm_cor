# 29 WITH - 꿈프 개발 착수 전 확인 사항

## 1. 확정된 개발 방향

현재 확정된 방향:

- 29 WITH는 새 프로젝트로 개발한다.
- 기술 스택은 프로젝트 목적에 맞게 새로 선정한다.
- 29초영화제/29역숏폼왕 출품 데이터는 DB 접근 방식으로 연동한다.

이 방향이면 관리자 페이지, 선생님 페이지, 메일, PDF, 외부 출품 데이터 동기화까지 한 시스템 안에서 안정적으로 설계할 수 있다.

## 4. 외부 DB 접근 관련 필요 정보

가장 중요하다.

29 WITH가 29초영화제/29역숏폼왕 출품 데이터를 DB로 가져오려면 아래 정보가 필요하다.

### 4.1 공통

필요 정보:

- DB 종류
- 접속 host
- port
- database name
- user
- password
- 접근 가능한 IP 제한 여부
- 읽기 전용 계정 제공 가능 여부
- 운영 DB와 복제/조회용 DB 분리 여부

권장:

- 운영 DB에 직접 붙기보다 읽기 전용 계정 또는 replica DB를 사용하는 것이 안전하다.
- 29 WITH 서버 IP를 허용 목록에 등록하는 방식이 좋다.
- DB 비밀번호는 화면에 원문 노출하지 않고 암호화 저장한다.

### 4.2 29초영화제 DB에서 필요한 데이터

필수 필드:

- 외부 행사 식별값
- 행사명
- 출품작 ID
- 소속명
- 작품명
- 감독명
- 출품 URL
- 출품일
- 예심평점
- 수상결과

본심 여부 기준:

```text
수상결과 = 1차통과 → 본심 진출
```

### 4.3 29역숏폼왕 DB에서 필요한 데이터

필수 필드:

- 외부 행사 식별값
- 행사명
- 출품작 ID
- 팀명
- 작품명
- 출품자명
- 출품 URL
- 출품일
- 평점

주의:

- 팀명 필드가 반드시 필요하다.
- 29역숏폼왕은 본심 여부를 표시하지 않는다.

### 4.4 행사 식별값 정책

행사명만 고유 ID로 사용하지 않는다.

이유:

- 행사명이 수정될 수 있다.
- 띄어쓰기나 표기 차이가 생길 수 있다.
- 같은 이름 규칙이 반복될 수 있다.

권장 기준:

```text
외부 행사 ID가 있으면 external_event_id로 저장
없으면 29 WITH에서 external_event_key 생성
```

외부 행사 ID가 없는 경우 묶어서 저장할 값:

- 행사명
- 연도
- 행사 유형
- 외부 관리자 페이지 URL
- 외부 DB 테이블 기준값

## 5. DB 동기화 방식 제안

외부 데이터가 변경될 수 있으므로 단순 덮어쓰기는 위험하다.

권장 방식:

```text
1. 외부 DB에서 행사별 출품 데이터 조회
2. external_submission_id 기준으로 기존 데이터와 비교
3. 신규/수정/삭제 가능 항목 구분
4. 큰 변화가 있으면 관리자 검토 대기
5. 완전 일치 항목만 자동 매칭
6. 유사 항목은 관리자 수동 매칭 후보로 표시
```

고유키 우선순위:

```text
1순위: 외부 출품작 ID
2순위: external_event_id + 출품 URL
3순위: external_event_id + 소속명/팀명 + 작품명 + 감독/출품자명 + 출품일
```

자동 매칭 기준:

```text
dream_event_id + external_event_id + participation_id + 출품 소속명/팀명 완전 일치
```

자동 매칭 금지:

- 공백 제거 후 일치
- 부분 일치
- 유사어 일치
- 학교명만 일치

이런 경우는 관리자 수동 매칭 후보로만 표시한다.

## 6. 개발 전 확인해야 할 운영 정보

### 6.1 관리자 계정

필요 정보:

- 메인 관리자 이메일
- 초기 비밀번호
- 관리자 추가/삭제 방식

정책:

- 비밀번호는 평문 저장 금지.
- 최초 로그인 후 변경 유도.
- 모든 관리자 작업 이력 저장.

### 6.2 메일 발송

필요 정보:

- 발송 방식: SMTP 또는 메일 발송 API
- 기본 발신자 이름
- 기본 발신자 이메일
- 행사별 발신자 변경 허용 여부
- 발송 도메인 인증 가능 여부

이미 확정된 정책:

- 자동 메일은 예약 후 관리자가 확인하고 발송한다.
- 발송일과 시간은 관리자가 수정할 수 있다.
- 수신자는 추가/수정할 수 있다.
- 같은 내용의 메일을 여러 선생님에게 한 번에 보낼 수 있다.
- 메일 템플릿은 추가/삭제/수정 가능하다.

### 6.3 파일 저장

필요 정보:

- 파일 저장 위치
- 최대 업로드 용량
- 허용 파일 형식
- 개인정보 파일 보관 기간

파일 종류:

- 포스터
- 통장사본
- 신분증
- 활동확인서 템플릿
- 학교별 활동확인서 파일
- 심사표 PDF
- 쿠폰 엑셀

### 6.4 활동확인서 템플릿

필요 정보:

- 기본 템플릿 파일 형식
- 자동 텍스트 배치가 필요한지
- 학교별 수동 파일 첨부를 허용할지

현재 정책:

- 템플릿 업로드 가능.
- 가능하면 텍스트 위치를 설정해 자동 PDF 생성.
- 자동 생성이 어렵다면 학교별 디자인 PNG/PDF 직접 첨부 가능.
- 학교명, 작품명, 학생명, 활동 기간, 발급일, 제호, 본문 문구는 관리자 수정 가능.

## 9. 현재 판단

기획과 정책은 개발 착수 가능한 수준이다.

다만 외부 DB 연동은 실제 DB 구조를 확인해야 정확히 구현할 수 있다.

따라서 개발은 아래 순서가 가장 안전하다.

```text
1. 29 WITH 기본 프로젝트 구축
2. 관리자/선생님 계정과 권한
3. 행사/신청/선정
4. 메일 템플릿/예약 메일
5. 외부 DB 연동 어댑터 구조
6. 실제 DB 정보 확인 후 동기화 구현
7. 활동확인서/심사표 자동화
```

외부 DB 정보가 늦게 들어와도, 먼저 내부 시스템 구조는 충분히 개발할 수 있다.
