# 29 WITH - 처음 들어온 꿈프 선생님 활동 동선 검토

## 1. 검토 목적

처음 들어온 선생님이 29 WITH에서 꿈프를 신청하고, 선정 후 학생 출품을 관리하고, 활동확인서/심사표/혜택까지 받는 과정이 자연스럽게 이어지는지 확인한다.

검토 기준:

- 선생님이 다음에 무엇을 해야 하는지 항상 알 수 있는가?
- 신청 상태, 선정 상태, 출품 상태가 끊기지 않고 이어지는가?
- 관리자 화면의 처리 결과가 선생님 화면에 정확히 반영되는가?
- 외부 29초영화제/29역숏폼왕 출품 정보와 꿈프 신청 정보가 충돌 없이 연결되는가?
- 오류나 예외 상황에서 선생님이 문의 또는 조치를 할 수 있는가?

---

## 2. 처음 들어온 선생님의 전체 동선

```text
회원가입
→ 로그인
→ 신청 가능한 행사 확인
→ 행사 상세 확인
→ 프로필 작성
→ 신청서 작성
→ 신청 완료 확인
→ 내 대시보드에서 신청 상태 확인
→ 선정 결과 확인
→ 선정 후 담당자 페이지 생성
→ 구독권 확인
→ 출품 소속명/팀명 확인
→ 학생 출품 진행
→ 학생/작품 관리에서 출품 확인
→ 미확인 작품 조치
→ 출품 마감
→ 관리자 최종 승인
→ 활동확인서 다운로드
→ 심사 결과 반영
→ 심사표 다운로드
→ 간식비 지급 상태 확인
→ 프로젝트 종료 후 참여 이력 확인
```

이 흐름은 크게 7단계로 나눈다.

1. 가입/신청
2. 신청 확인/선정 대기
3. 선정 후 준비
4. 출품 진행
5. 출품 마감 후 확인
6. 결과/문서 수령
7. 종료/히스토리

---

## 3. 1단계: 가입/신청

### 3.1 선생님 행동

```text
꿈나무양성프로젝트 유형으로 회원가입
→ 로그인
→ 신청 가능한 행사 목록 확인
→ 행사 상세 확인
→ 신청하기
```

### 3.2 필요한 화면

- 회원가입
- 신청 가능한 행사
- 행사 상세
- 프로필 작성
- 신청서 작성
- 신청 완료

### 3.3 연결되어야 하는 데이터

- 회원 유형: 꿈나무양성프로젝트
- 이름
- 이메일
- 휴대폰번호
- 비밀번호
- 학교명
- 학교급
- 담당교사명
- 담당과목
- 연락처
- 출품 소속명/팀명
- 통장사본
- 신분증
- 신청 행사
- 예상 출품편수
- 활용 계획
- 출품 예정일

### 3.4 오류 가능 지점

#### 문제 1. 회원가입 후 바로 무엇을 해야 하는지 모를 수 있음

대응:

- 로그인 후 신청 가능한 행사가 있으면 `신청 가능한 행사` 화면으로 바로 안내한다.
- 신청 가능한 행사가 없으면 “현재 신청 가능한 꿈프가 없습니다” 빈 상태를 보여준다.

#### 문제 2. 학교명과 출품 소속명/팀명을 혼동할 수 있음

대응:

- 신청서에서 `학교명`과 `출품 소속명/팀명`을 분리해서 보여준다.
- 출품 소속명/팀명 입력란 근처에 짧고 강한 안내를 둔다.

권장 문구:

```text
출품 소속명/팀명은 실제 출품 시 입력하는 이름과 띄어쓰기까지 동일해야 합니다.
```

#### 문제 3. 신청 완료 여부를 불안해할 수 있음

대응:

- 제출 후 반드시 `신청 완료` 화면을 보여준다.
- 내 대시보드 상단에 `신청 접수 완료`를 표시한다.
- 확인 메일을 발송하고 공지/메일함에도 남긴다.

### 3.5 검토 결과

현재 계획상 흐름은 가능하다.

다만 `신청 가능한 행사 → 행사 상세 → 신청서 작성 → 신청 완료` 화면이 실제 개발에서 반드시 하나의 흐름으로 연결되어야 한다.

---

## 4. 2단계: 신청 확인/선정 대기

### 4.1 선생님 행동

```text
내 대시보드 진입
→ 신청 접수 상태 확인
→ 선정 결과 대기
→ 신청서 다시 보기
```

### 4.2 필요한 화면

- 내 대시보드
- 참여 프로젝트
- 신청서 보기
- 공지/메일함

### 4.3 상태

- 신청서 작성 전
- 신청 임시저장
- 신청 접수 완료
- 관리자 검토 중
- 선정 완료
- 예비
- 미선정
- 신청 취소

### 4.4 오류 가능 지점

#### 문제 1. 신청 후 대시보드에 아무것도 없으면 불안함

대응:

- 신청 완료 즉시 대시보드에 프로젝트 카드를 생성한다.
- 선정 전이라도 “관리자 검토 중” 상태를 보여준다.

#### 문제 2. 미선정/예비 상태의 다음 행동이 모호함

대응:

- 예비: “추가 선정 시 메일과 대시보드로 안내됩니다.”
- 미선정: “이번 프로젝트에는 선정되지 않았습니다. 다음 모집 시 다시 신청할 수 있습니다.”

#### 문제 3. 과거 참여 이력 때문에 불안할 수 있음

대응:

- 대시보드에 참여 신뢰 상태를 짧게 표시한다.
- 자세한 내용은 `참여 이력`에서 확인하게 한다.
- “패널티” 대신 “확인 필요 이력 있음”, “우선순위 조정 가능”으로 표현한다.

### 4.5 검토 결과

선정 전 상태가 대시보드에 반드시 보여야 한다.

이 부분이 빠지면 선생님은 신청이 누락된 것으로 느낄 수 있다.

---

## 5. 3단계: 선정 후 준비

### 5.1 선생님 행동

```text
선정 메일 확인
→ 내 대시보드 확인
→ 구독권 확인
→ 출품 소속명/팀명 확인
→ 출품 일정 확인
```

### 5.2 관리자 액션

```text
선정 처리
→ 선정 안내 메일 발송
→ 한국경제신문 구독권 지급
→ 담당자 페이지 생성
```

### 5.3 선생님 화면 반영

- 선정 상태: 선정 완료
- 구독권: 지급 완료 또는 지급 예정
- 학생/작품 관리: 신청 작품 수만큼 슬롯 생성
- 공지/메일함: 선정 안내 저장
- 출품 소속명/팀명: 복사 가능한 형태로 표시

### 5.4 오류 가능 지점

#### 문제 1. 선정은 되었는데 작품 관리 화면이 비어 있을 수 있음

대응:

- 선정 처리와 동시에 신청 작품 수만큼 작품 슬롯을 생성한다.
- 작품 슬롯 상태는 처음에 `출품 전` 또는 `확인 대기`로 둔다.

#### 문제 2. 구독권 지급 실패 또는 쿠폰 부족

대응:

- 관리자 화면에서 쿠폰 부족을 먼저 경고한다.
- 선생님 화면에는 `지급 예정`으로 표시하고 내부 오류는 노출하지 않는다.

#### 문제 3. 출품 소속명/팀명을 잊을 수 있음

대응:

- 내 대시보드와 학생/작품 관리 상단에 항상 표시한다.
- 복사 버튼을 제공한다.

### 5.5 검토 결과

선정 처리 후 자동으로 생성되어야 하는 항목이 많다.

개발 시 `선정 처리` 버튼은 단순 상태 변경이 아니라 아래 작업을 묶어서 처리해야 한다.

```text
상태 변경
메일 발송
쿠폰 지급
담당자 페이지 생성
작품 슬롯 생성
공지/메일함 기록
```

## 6. 4단계: 출품 진행

### 6.1 선생님 행동

```text
학생들이 29초영화제/29역숏폼왕에 출품
→ 선생님이 29 WITH에서 출품 확인 상태 확인
→ 미확인 작품 있으면 학생에게 확인
→ 필요 시 작품 링크 직접 입력 또는 관리자에게 확인 요청
```

### 6.2 자동 처리

```text
D-14 안내 메일
D-10 안내 메일
D-5 안내 메일
D-1 안내 메일
외부 관리자 페이지 출품작 자동 동기화
학교명/소속명/팀명 기준 자동 매칭
```

### 6.3 연결 기준

29초영화제:

```text
꿈프 학교명 또는 출품 소속명 = 29초영화제 출품작 관리의 소속명
```

29역숏폼왕:

```text
꿈프 학교명 또는 출품 소속명 = 29역숏폼왕 출품작 관리의 팀명
```

공통:

```text
행사 식별값 + 소속명/팀명 완전 일치
```

### 6.4 오류 가능 지점

#### 문제 1. 같은 학교가 여러 행사에 동시에 참여할 수 있음

대응:

- 매칭 기준에 반드시 행사 식별값을 포함한다.
- 학교명만으로 전체 출품작을 검색하면 안 된다.

필수 기준:

```text
external_event_id + participation_id + school_affiliation_name
```

#### 문제 2. 띄어쓰기 차이로 자동 매칭 실패

대응:

- 자동 매칭은 완전 일치만 인정한다.
- 실패 시 선생님에게 입력명 확인 안내를 보여준다.
- 관리자는 수동 매칭할 수 있어야 한다.

#### 문제 3. 외부 연동이 실패했는데 선생님이 불안해할 수 있음

대응:

- 선생님 화면에는 “관리자 확인 중”으로 표시한다.
- 관리자 화면에는 연동 실패 원인과 재시도 버튼을 제공한다.

#### 문제 4. 신청 작품 수보다 많이 출품한 경우

대응:

- 관리자 화면에는 초과 출품으로 표시한다.
- 선생님 화면에는 “추가 확인된 작품”으로 보여준다.
- 활동확인서 포함 여부는 관리자 최종 승인에서 결정한다.

#### 문제 5. 신청 작품 수보다 적게 출품한 경우

대응:

- 선생님 대시보드에 미확인 수를 표시한다.
- D-day 안내와 함께 조치 버튼을 보여준다.
- 마감 후에도 미확인인 경우 관리자 확인 대상으로 넘긴다.

### 6.5 검토 결과

출품 단계에서 가장 중요한 것은 `행사 식별값 + 정확한 소속명/팀명`이다.

이 기준이 지켜지면 A영화제와 B영화제가 겹쳐도 잘못 가져올 가능성을 줄일 수 있다.

## 7. 5단계: 출품 마감 후 확인

### 7.1 선생님 행동

```text
출품 마감 후 대시보드 확인
→ 관리자 최종 확인 대기
→ 출품 완료 작품 수 확인
```

### 7.2 관리자 액션

```text
최종 출품작 리스트 확인
→ 학교별 승인
→ 활동확인서 생성 가능 상태로 변경
→ 발급 안내 메일 발송
```

### 7.3 오류 가능 지점

#### 문제 1. 마감 후 선생님이 무엇을 기다리는지 모를 수 있음

대응:

- 선생님 화면에 `관리자 최종 확인 중` 상태를 표시한다.

문구:

```text
출품 마감이 종료되었습니다.
관리자 최종 확인 후 활동확인서를 다운로드할 수 있습니다.
```

#### 문제 2. 관리자가 승인 전인데 다운로드 버튼이 보일 수 있음

대응:

- 활동확인서 다운로드 버튼은 승인 후에만 노출한다.
- 승인 전에는 상태 문구만 보여준다.

#### 문제 3. 최종 승인 후 메일 발송이 누락될 수 있음

대응:

- 승인 버튼 동작에 발급 안내 메일 발송 예약 또는 즉시 발송을 포함한다.
- 메일 실패 시 관리자 작업함에 남긴다.

### 7.4 검토 결과

출품 마감 후에는 선생님이 직접 할 일이 줄어든다.

따라서 화면은 “기다리는 상태”를 명확히 알려줘야 한다.

## 8. 6단계: 결과/문서 수령

### 8.1 선생님 행동

```text
활동확인서 다운로드
→ 수상작 발표 후 심사표 확인
→ 심사표 PDF 다운로드
→ 간식비 지급 상태 확인
```

### 8.2 관리자 액션

```text
활동확인서 PDF 생성
→ 발급 안내 메일 발송
→ 심사 결과 동기화
→ 순위/본심 여부 반영
→ 심사표 PDF 생성
→ 결과 반영 안내 메일 발송
→ 간식비 지급 예정일 입력
```

### 8.3 오류 가능 지점

#### 문제 1. 활동확인서와 심사표 제공 시점이 다름

대응:

- 두 문서를 같은 화면에 묶지 않는다.
- 활동확인서와 심사표 메뉴를 분리한다.
- 혜택/지원에는 요약 상태만 표시한다.

#### 문제 2. 심사표 반영 전 빈 화면처럼 보일 수 있음

대응:

- `수상작 발표 후 제공됩니다` 상태를 표시한다.

#### 문제 3. 간식비 지급일이 바뀔 수 있음

대응:

- 관리자 화면에서 지급 예정일 변경 이력을 남긴다.
- 선생님 화면에는 최신 예정일과 안내 문구만 보여준다.
- 변경 시 공지/메일함에 안내를 남긴다.

### 8.4 검토 결과

활동확인서, 심사표, 간식비는 제공 시점이 다르다.

선생님 대시보드에서는 세 항목을 요약으로 보여주되, 상세 화면은 분리해야 혼란이 적다.

---

## 9. 7단계: 종료/히스토리

### 9.1 선생님 행동

```text
종료된 프로젝트 확인
→ 활동확인서/심사표 다시 다운로드
→ 참여 이력 확인
```

### 9.2 관리자 액션

```text
종료 전 점검
→ 미완료 항목 확인
→ 꿈프 종료
→ 히스토리 저장
→ 학교별 참여 이력 반영
```

### 9.3 오류 가능 지점

#### 문제 1. 종료 후 문서를 다시 받을 수 없으면 불편함

대응:

- 선생님 참여 프로젝트의 종료 영역에 다운로드 버튼을 유지한다.

#### 문제 2. 종료 후 참여 이력이 반영되지 않으면 다음 선정 기준이 흔들림

대응:

- 종료 버튼 실행 시 학교별 참여 이력을 자동 업데이트한다.

반영 항목:

- 신청 횟수
- 선정 횟수
- 출품 완료 횟수
- 미이행 횟수
- 성실 참여 여부
- 확인 필요 이력 여부

#### 문제 3. 미완료 상태인데 종료할 수 있음

대응:

- 종료 전 체크리스트를 통과해야 종료 가능하다.

체크 항목:

- 최종 출품 리스트 승인
- 활동확인서 발급 상태
- 심사표 발급 또는 미대상 처리
- 간식비 상태 입력
- 구독권 지급 상태
- 미이행 학교 반영

### 9.4 검토 결과

종료는 단순 상태 변경이 아니라 히스토리와 참여 이력 반영까지 포함해야 한다.

## 10. 기능 간 연계 체크리스트

### 10.1 신청 → 대시보드

- 신청 완료 즉시 대시보드에 프로젝트 카드가 생성되는가?
- 신청 접수 완료 시간이 표시되는가?
- 신청 완료 메일이 공지/메일함에 남는가?

### 10.2 선정 → 담당자 페이지

- 선정 처리 시 담당자 페이지가 자동 생성되는가?
- 신청 작품 수만큼 작품 슬롯이 생성되는가?
- 구독권 지급 상태가 혜택/지원에 표시되는가?
- 선정 메일이 공지/메일함에 남는가?

### 10.3 외부 출품 → 작품 관리

- 행사 식별값 기준으로 해당 행사 출품작만 가져오는가?
- 소속명/팀명 완전 일치 기준으로 자동 매칭되는가?
- 미매칭 작품은 관리자와 선생님에게 각각 적절히 표시되는가?
- 수동 매칭 이력이 남는가?

### 10.4 출품 마감 → 활동확인서

- 출품 마감 후 관리자 승인 전에는 다운로드 버튼이 숨겨지는가?
- 승인 후 PDF 생성 버튼이 활성화되는가?
- PDF 생성 후 선생님에게 안내 메일이 발송되는가?

### 10.5 수상 발표 → 심사표

- 심사 결과 동기화 후 순위가 학교별 작품 목록에 반영되는가?
- 29초영화제만 본심 여부가 표시되는가?
- 29역숏폼왕에서는 본심 열이 숨겨지는가?
- 결과 반영 후 안내 메일이 발송되는가?

### 10.6 간식비 → 선생님 대시보드

- 관리자가 지급 예정일을 입력하면 선생님 대시보드에 바로 반영되는가?
- 지급 예정일 변경 시 최신 정보만 노출되는가?
- 보류 또는 미대상 사유가 짧게 표시되는가?

### 10.7 종료 → 히스토리/참여 이력

- 종료 전 체크리스트가 있는가?
- 종료 후 진행 중 목록에서 빠지는가?
- 히스토리에 보관되는가?
- 선생님은 종료 후에도 문서를 다운로드할 수 있는가?
- 학교별 참여 이력이 업데이트되는가?

## 11. 예상되는 주요 오류와 방지 방법

| 오류 | 원인 | 방지 방법 |
|---|---|---|
| 신청했는데 대시보드에 안 보임 | 신청과 프로젝트 카드 생성이 분리됨 | 신청 완료 시 프로젝트 카드 자동 생성 |
| 선정됐는데 작품 슬롯이 없음 | 선정 처리 후 후속 생성 누락 | 선정 처리에 슬롯 생성 포함 |
| 다른 행사 출품작이 섞임 | 학교명만으로 매칭 | 행사 식별값 + 소속명/팀명 기준 |
| 자동 매칭 실패 | 띄어쓰기/표기 차이 | 완전 일치 안내, 수동 매칭, 확인 요청 |
| 선생님이 오류를 봄 | 외부 연동 실패 노출 | 선생님 화면에는 관리자 확인 중으로 표시 |
| 확인서 다운로드가 너무 일찍 보임 | 승인 상태 미분리 | 승인 후 다운로드 버튼 노출 |
| 심사표에 본심 열이 잘못 표시됨 | 행사 유형 미반영 | 29초영화제만 본심 표시 |
| 종료 후 문서 접근 불가 | 히스토리와 선생님 종료 목록 미연결 | 종료 프로젝트 다운로드 유지 |

## 12. 최종 판단

처음 들어온 선생님 동선은 전체적으로 구현 가능하다.

다만 아래 5가지는 개발 시 반드시 하나의 연결 흐름으로 처리해야 오류가 적다.

1. 신청 완료 → 대시보드 프로젝트 카드 생성
2. 선정 처리 → 담당자 페이지, 작품 슬롯, 쿠폰, 메일 동시 생성
3. 외부 출품 동기화 → 행사 식별값 + 소속명/팀명 기준 매칭
4. 출품 최종 승인 → 활동확인서 PDF와 안내 메일 연결
5. 꿈프 종료 → 히스토리 저장과 참여 이력 반영

선생님 화면에서는 모든 단계에서 “현재 상태”와 “다음 행동”이 보여야 한다.

관리자 화면에서는 모든 단계에서 “처리 필요 항목”과 “다음 처리 버튼”이 보여야 한다.
