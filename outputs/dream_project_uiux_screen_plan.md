# 29 WITH - 꿈프 페이지 및 화면/버튼 UIUX 계획

## 1. 전체 설계 방향

29 WITH의 꿈프 화면은 두 종류로 나눈다.

- 관리자 화면: 여러 행사와 여러 학교를 빠르게 확인하고 처리하는 화면
- 선생님 화면: 우리 학교의 신청, 선정, 출품, 혜택, 문서 상태를 안심하고 확인하는 화면

핵심 원칙:

- 첫 화면에는 “지금 처리해야 할 일”을 가장 먼저 보여준다.
- 긴 설명보다 상태, 숫자, 버튼을 우선한다.
- 한 화면에 모든 기능을 넣지 않고, 운영 단계별로 나눈다.
- 버튼은 사용자가 지금 할 수 있는 행동만 노출한다.
- 완료된 단계는 접어두고, 문제가 있는 항목만 눈에 띄게 보여준다.
- 관리자용 내부 정보는 선생님 화면에 노출하지 않는다.

---

## 2. 관리자 화면 구조

관리자 상단 카테고리:

```text
전체
꿈프
서포터즈
명예홍보대사
파트너사
```

`꿈프` 선택 후 왼쪽 메뉴:

```text
꿈프 대시보드
행사 운영
회원/참여 학교
신청/선정
출품 확인
활동확인서
심사 결과
혜택/지원
메일/공지
연동 설정
히스토리
```

왼쪽 메뉴 하단에는 진행 중인 연결 행사를 표시한다.

```text
진행 중인 꿈프
- 제13회 박카스 29초영화제
- 제1회 동반성장 29역숏폼왕
- 간단요리사 29역숏폼왕
```

행사를 선택하면 모든 꿈프 메뉴가 해당 행사 기준으로 바뀐다.

## 3. 관리자 - 꿈프 대시보드

목적:

- 현재 운영 중인 꿈프 전체 상태 확인
- 어떤 행사를 먼저 처리해야 하는지 판단
- 지연, 실패, 승인 대기 건을 빠르게 확인

상단 구성:

- 진행 중인 행사 카드
- 처리 필요 건수
- 출품 확인 진행률
- 마감 임박 행사

행사 카드 표시 정보:

- 행사명
- 행사 유형
- 모집/출품/마감 상태
- 선정 학교 수
- 신청 작품 수
- 확인된 출품 수
- 처리 필요 건수

주요 버튼:

- `행사 보기`
- `처리 필요 보기`
- `출품 확인`

숨길 정보:

- 외부 연동 상세 로그
- 긴 정책 설명
- 다른 카테고리 상세 정보

UX:

- 대시보드는 보고서가 아니라 작업 출발점이다.
- 카드 클릭 시 해당 행사 운영 화면으로 이동한다.
- 처리 필요 건수가 있는 행사를 시각적으로 먼저 보이게 한다.

---

## 4. 관리자 - 행사 운영

목적:

- 행사 기본 정보와 꿈프 운영 상태를 관리
- 모집 시작부터 종료까지 단계별 진행

표시 정보:

- 행사명
- 행사 유형
- 공모기간
- 출품 마감일
- 모집 목표 작품 수
- 현재 신청 작품 수
- 현재 출품 확인 수
- 행사 상태
- 꿈프 운영 상태

주요 버튼:

- `행사 정보 수정`
- `꿈프 모집 시작`
- `모집 안내 메일 발송`
- `외부 연동 확인`
- `꿈프 종료`

버튼 노출 조건:

- `꿈프 모집 시작`: 모집 전 상태에서만 활성화
- `모집 안내 메일 발송`: 모집 시작 후 활성화
- `외부 연동 확인`: 연동 정보 입력 후 활성화
- `꿈프 종료`: 활동확인서와 심사표까지 완료 후 활성화

UX:

- 운영 단계는 체크리스트 형태로 보여준다.
- 완료된 단계는 작게 접고, 다음 단계 버튼을 크게 보여준다.
- `꿈프 종료`는 실수 방지를 위해 확인 팝업을 거친다.

## 5. 관리자 - 회원/참여 학교

목적:

- 꿈프 회원으로 가입한 선생님과 학교 프로필 확인
- 학교별 참여 이력과 관리자 메모 관리

표시 방식:

- 학교 단위 리스트
- 학교 선택 시 오른쪽 상세 패널

리스트 표시 정보:

- 학교명
- 담당교사
- 연락처
- 이메일
- 최근 참여 상태
- 신청 횟수
- 선정 횟수
- 출품 완료 횟수
- 미이행 횟수

상세 패널 표시 정보:

- 프로필 정보
- 출품 소속명/팀명
- 참여 이력
- 관리자 메모
- 증빙 업로드 상태

주요 버튼:

- `프로필 보기`
- `참여 이력 보기`
- `메모 추가`
- `학교명 변경 승인`
- `메일 보내기`

UX:

- 리스트에는 판단에 필요한 최소 정보만 표시한다.
- 긴 이력과 메모는 상세 패널에서 확인한다.
- 관리자 메모는 학교 단위로 누적된다.

---

## 6. 관리자 - 신청/선정

목적:

- 행사별 신청 학교 검토
- 선정, 예비, 미선정 처리
- 쿠폰 지급과 담당자 페이지 생성

표시 방식:

- 학교 단위 리스트
- 복수 선택 가능
- 선택 학교 상세 패널

리스트 표시 정보:

- 신청일
- 학교명
- 담당교사
- 출품 소속명/팀명
- 예상 출품편수
- 출품 예정일
- 참여 신뢰 상태
- 상태

상태:

- 신청 접수
- 검토 중
- 선정
- 예비
- 미선정

주요 버튼:

- `선정`
- `예비`
- `미선정`
- `선정 메일 발송`
- `메일 재발송`
- `엑셀 다운로드`

버튼 동작:

- `선정`: 신청 상태 변경, 선정 메일 발송, 구독권 지급, 담당자 페이지 생성
- `예비`: 상태만 예비로 변경하고 예비 안내 메일 발송
- `미선정`: 상태 변경 후 미선정 안내 메일 발송

UX:

- 선정 버튼은 영향이 크므로 처리 전 확인 팝업을 띄운다.
- 쿠폰 부족 시 선정 처리 전 경고한다.
- 예상 출품편수 합계가 목표 작품 수를 초과해도 신청은 유지한다.

---

## 7. 관리자 - 출품 확인

목적:

- 외부 29초영화제/29역숏폼왕 관리자 페이지에서 가져온 출품작을 학교별로 확인
- 자동 매칭 실패 건을 빠르게 조치

표시 방식:

- 학교 단위 요약 리스트
- 학교 선택 시 작품 상세 리스트

학교 리스트 표시 정보:

- 학교명
- 출품 소속명/팀명
- 신청 작품 수
- 자동 확인 수
- 직접 확인 수
- 미확인 수
- 매칭 상태
- 마지막 연동 시간

작품 상세 표시 정보:

- 작품명
- 감독명 또는 출품자명
- 출품 URL
- 출품일
- 매칭 방식
- 확인 상태

주요 버튼:

- `전체 동기화`
- `선택 학교 재확인`
- `수동 매칭`
- `작품 링크 추가`
- `확인 완료 처리`
- `선생님에게 확인 요청`

UX:

- 작품이 많으므로 첫 화면은 학교 단위로 보여준다.
- 미확인 수가 있는 학교만 필터링할 수 있어야 한다.
- 외부 연동 오류 상세는 별도 패널에서만 보여준다.
- 학교명 완전 일치 원칙은 안내 영역이 아니라 문제 발생 시에만 노출한다.

---

## 8. 관리자 - 활동확인서

목적:

- 출품 마감 후 최종 출품작 리스트 승인
- 활동확인서 PDF 생성
- 선생님에게 발급 안내 메일 발송

표시 방식:

- 학교 단위 리스트
- 승인 전/승인 완료 탭

리스트 표시 정보:

- 학교명
- 최종 확인 작품 수
- 학생 이름
- 작품명
- 확인서 상태
- 승인 상태

주요 버튼:

- `최종 리스트 확인`
- `승인`
- `PDF 미리보기`
- `활동확인서 생성`
- `발급 메일 발송`
- `재발급`

버튼 노출 조건:

- `승인`: 출품 마감 후 활성화
- `활동확인서 생성`: 승인 후 활성화
- `발급 메일 발송`: PDF 생성 후 활성화

UX:

- 승인 버튼은 학교별로 두되, 복수 승인도 가능해야 한다.
- 승인 전에는 PDF 다운로드 버튼을 보여주지 않는다.
- PDF 미리보기에서 학교명, 작품명, 학생 이름, 기간을 확인할 수 있어야 한다.

---

## 9. 관리자 - 심사 결과

목적:

- 수상작 발표 후 외부 관리자 페이지의 점수/수상결과를 반영
- 학교별 순위와 본심 진출 여부를 제공
- 심사표 PDF 생성

표시 정보:

- 학교명
- 작품명
- 감독명 또는 출품자명
- 순위
- 본심 진출 여부
- 반영 상태

주요 버튼:

- `심사 결과 동기화`
- `순위 계산`
- `본심 여부 반영`
- `PDF 미리보기`
- `심사표 생성`
- `결과 안내 메일 발송`

UX:

- 원본 점수는 관리자만 확인하되, 선생님 화면에는 순위와 본심 여부 중심으로 제공한다.
- 29역숏폼왕은 본심 항목을 숨긴다.
- 결과 반영 후 선생님에게 안내 메일을 발송한다.

---

## 10. 관리자 - 혜택/지원

목적:

- 구독권 쿠폰, 간식비, 기타 지원 상태 관리
- 선생님 화면에 혜택 지급 상태 노출

표시 정보:

- 학교명
- 구독권 지급 상태
- 쿠폰번호
- 간식비 지급 상태
- 간식비 지급 예정일
- 지급 금액
- 안내 메모

주요 버튼:

- `쿠폰 업로드`
- `쿠폰 지급`
- `간식비 정보 입력`
- `지급 완료 처리`
- `지원 안내 메일 발송`

간식비 상태:

- 검토 중
- 지급 예정
- 지급 완료
- 보류
- 지급 대상 아님

UX:

- 선생님에게 보여줄 문구를 관리자 입력창에서 바로 확인할 수 있게 한다.
- 내부 정산 정보와 선생님 노출 정보는 분리한다.
- 지급 예정일이 입력되면 선생님 대시보드에 자동 표시한다.

---

## 11. 관리자 - 메일/공지

목적:

- 자동 메일 발송 현황 확인
- 발송 예정 메일의 내용, 발송일, 수신자를 확인하고 수정
- 메일 템플릿 관리
- 실패 건 조치
- 선생님에게 공지 재전송

표시 정보:

- 메일 유형
- 대상 행사
- 대상 학교
- 수신자 이메일
- 메일 제목
- 발송 예정일
- 발송일
- 발송 상태
- 재발송 여부

주요 버튼:

- `메일 템플릿 수정`
- `예약 메일 보기`
- `수신자 확인`
- `메일 미리보기`
- `발송일 수정`
- `수신자 수정`
- `즉시 발송`
- `재발송`
- `선택 대상 발송`
- `공지 등록`

UX:

- D-14, D-10, D-5, D-1 안내 메일은 자동 예약 상태를 보여준다.
- 예약된 메일은 발송 전까지 제목, 본문, 발송일, 수신자를 수정할 수 있어야 한다.
- 자동 메일도 발송 전에는 미리보기와 테스트 발송이 가능해야 한다.
- 실패 건은 한 화면에서 다시 보낼 수 있어야 한다.
- 선생님 화면에는 발송 실패 로그를 노출하지 않는다.

메일 유형:

- 모집 안내
- 신청 완료 안내
- 선정 안내
- 예비 안내
- 미선정 안내
- 출품 마감 D-14 안내
- 출품 마감 D-10 안내
- 출품 마감 D-5 안내
- 출품 마감 D-1 안내
- 출품 확인 요청
- 출품 확인 완료 안내
- 활동확인서 발급 안내
- 심사 결과 반영 안내
- 간식비/지원 안내
- 꿈프 종료 안내

메일 관리 화면 탭:

```text
템플릿
예약 메일
발송 이력
실패/재발송
수신자 목록
```

### 11.1 메일 템플릿

관리자가 자동 메일의 기본 제목과 본문을 관리한다.

표시 정보:

- 메일 유형
- 기본 제목
- 기본 본문
- 사용 변수
- 마지막 수정일
- 수정자

사용 변수 예시:

- 행사명
- 학교명
- 담당교사명
- 신청 작품 수
- 출품 예정일
- 출품 URL
- 출품 소속명/팀명
- 쿠폰번호
- 활동확인서 다운로드 링크
- 심사표 다운로드 링크
- 간식비 지급 예정일

주요 버튼:

- `템플릿 수정`
- `미리보기`
- `테스트 발송`
- `기본값 복원`

정책:

- 템플릿 수정은 이후 생성되는 예약 메일에 적용한다.
- 이미 예약된 메일에 적용하려면 관리자가 `예약 메일에 반영`을 선택해야 한다.

### 11.2 예약 메일

자동으로 발송될 메일을 발송 전에 확인하고 수정하는 화면이다.

표시 정보:

- 메일 유형
- 대상 행사
- 대상 학교
- 수신자 이메일
- 제목
- 발송 예정일
- 예약 상태

주요 버튼:

- `내용 수정`
- `발송일 수정`
- `수신자 수정`
- `미리보기`
- `테스트 발송`
- `즉시 발송`
- `예약 취소`

정책:

- 발송 전 예약 메일은 제목, 본문, 발송 예정일, 수신자 수정이 가능하다.
- 발송 완료된 메일은 수정할 수 없고 재발송만 가능하다.
- 수신자를 수정하면 수정 이력이 남아야 한다.
- 발송 예정일을 변경하면 기존 자동 발송 스케줄도 함께 변경된다.

### 11.3 수신자 목록

메일 발송 대상자를 확인하고 수정한다.

표시 정보:

- 학교명
- 담당교사명
- 기본 이메일
- 추가 수신 이메일
- 수신 제외 여부
- 최근 발송 상태

주요 버튼:

- `이메일 수정`
- `추가 수신자 등록`
- `수신 제외`
- `수신자 엑셀 다운로드`

정책:

- 기본 이메일은 회원 이메일이다.
- 추가 수신자는 행사 단위로 등록할 수 있다.
- 수신 제외는 자동 메일에서 제외하되, 관리자 화면에는 제외 상태를 표시한다.
- 이메일 수정 이력은 작업 이력에 남긴다.

### 11.4 발송 이력과 재발송

발송 완료 또는 실패한 메일을 확인한다.

표시 정보:

- 메일 유형
- 대상 학교
- 수신 이메일
- 발송일
- 발송 상태
- 열람 여부
- 재발송 횟수

주요 버튼:

- `메일 보기`
- `재발송`
- `다른 이메일로 재발송`

정책:

- 재발송은 원본 메일 내용을 기준으로 하되, 발송 전 수정 가능하다.
- 다른 이메일로 재발송한 경우 별도 이력이 남아야 한다.
- 실패 상태는 관리자에게만 표시한다.

## 15. 선생님 - 내 대시보드

목적:

- 신청이 잘 들어갔는지 확인

- 내부 상태와 오류 조치 버튼 제공

선생님 화면:

공통:

## 23. 재검토 결과: 보완이 필요한 지점

현재 계획의 큰 방향은 맞지만, 실제 운영 관점에서는 아래 항목이 추가로 필요하다.

### 23.1 선생님 신청 전 화면이 부족함

기존 계획은 신청 이후의 선생님 화면에 집중되어 있다.

하지만 선생님은 먼저 아래 과정을 거친다.

```text
행사 조회 → 상세 확인 → 신청 가능 여부 확인 → 프로필 작성 → 신청서 제출 → 접수 확인
```

따라서 선생님 화면에 `신청 가능한 행사` 화면을 추가해야 한다.

필요 화면:

- 신청 가능한 꿈프
- 행사 상세
- 프로필 작성/확인
- 신청서 작성
- 신청 완료 확인

핵심 버튼:

- `행사 자세히 보기`
- `신청하기`
- `프로필 저장`
- `신청서 제출`
- `신청 완료 확인`

UX:

- 신청 가능 상태가 아니면 `신청하기` 버튼은 비활성화한다.
- 신청 완료 후에는 “신청이 접수되었습니다” 화면을 반드시 보여준다.
- 이미 신청한 행사는 `신청 완료`로 표시하고 중복 신청을 막는다.

### 23.2 관리자 권한과 승인 이력이 부족함

현재 계획에는 관리자 화면이 있지만, 누가 어떤 작업을 했는지 남기는 구조가 부족하다.

추가 필요:

- 관리자 권한 구분
- 작업 이력
- 상태 변경 이력
- 메일 재발송 이력
- 수동 매칭 이력
- PDF 재발급 이력

권한 예시:

- 최고 관리자
- 꿈프 운영 관리자
- 메일/공지 관리자
- 혜택/쿠폰 관리자
- 조회 전용 관리자

UX:

- 위험 버튼은 권한이 있는 관리자에게만 보인다.
- 상태 변경 전 확인 팝업에서 변경 결과를 명확히 보여준다.
- 학교명 변경, 출품 소속명 변경, 최종 승인, 꿈프 종료는 이력을 반드시 남긴다.

### 23.3 활동확인서 템플릿 관리 화면이 필요함

활동확인서는 관리자가 샘플을 업로드하고 텍스트 위치를 맞춰야 한다.

추가 화면:

- 확인서 템플릿 관리

표시 정보:

- 템플릿명
- 적용 행사 유형
- 사용 여부
- 마지막 수정일
- 미리보기

주요 버튼:

- `템플릿 업로드`
- `텍스트 위치 설정`
- `샘플 PDF 생성`
- `기본 템플릿으로 지정`

UX:

- 빨간 표시 같은 내부 가이드 요소는 실제 발급 PDF에 포함되지 않아야 한다.
- 행사명, 학교명, 작품명, 학생 이름, 활동 기간, 발급일, 제호가 어디에 들어가는지 미리보기에서 확인할 수 있어야 한다.

### 23.4 쿠폰/간식비 재고와 지급 실패 처리가 부족함

혜택/지원 화면에는 지급 상태가 있지만, 실패 상황 처리가 더 필요하다.

추가 필요:

- 쿠폰 잔여 수량
- 쿠폰 중복 지급 방지
- 쿠폰 부족 경고
- 간식비 지급 보류 사유
- 계좌/증빙 미비 안내
- 지급 예정일 변경 이력

주요 버튼:

- `쿠폰 재고 보기`
- `미지급 학교 보기`
- `간식비 보류 사유 입력`
- `지급 예정일 일괄 입력`
- `지급 안내 재발송`

UX:

- 선생님에게는 내부 재고나 지급 실패 로그를 보여주지 않는다.
- 선생님에게 필요한 경우에는 “추가 확인 필요”와 조치 방법만 보여준다.

### 23.5 문의/요청 흐름이 부족함

선생님이 문제를 발견했을 때 바로 조치할 방법이 필요하다.

추가 화면:

- 문의/요청

요청 유형:

- 신청 상태 문의
- 학교명 변경 요청
- 출품 소속명/팀명 변경 요청
- 작품 확인 요청
- 활동확인서 재발급 요청
- 심사표 문의
- 간식비/지원 문의

선생님 버튼:

- `관리자에게 확인 요청`
- `학교명 변경 요청`
- `작품 확인 요청`
- `재발급 요청`

관리자 버튼:

- `요청 확인`
- `승인`
- `반려`
- `답변 등록`
- `처리 완료`

UX:

- 채팅형 AI 명령보다 먼저 정해진 요청 유형과 버튼을 제공하는 것이 안정적이다.
- 자유 입력은 보조로 두고, 관리자가 빠르게 처리할 수 있도록 요청 유형을 구조화한다.

### 23.6 종료 체크리스트가 더 구체적이어야 함

꿈프 종료 버튼은 단순 버튼이 아니라 최종 검수 흐름이어야 한다.

종료 전 체크 항목:

- 모든 신청 상태 확정
- 최종 출품작 리스트 승인
- 활동확인서 발급 완료
- 심사표 발급 완료 또는 미대상 처리
- 간식비 지급 상태 입력
- 구독권 지급 상태 확인
- 미이행 학교 반영
- 메일 발송 완료
- 히스토리 저장 확인

주요 버튼:

- `종료 전 점검`
- `미완료 항목 보기`
- `꿈프 종료`
- `히스토리로 이동`

UX:

- 미완료 항목이 있으면 종료 버튼을 비활성화하거나 강한 경고를 표시한다.
- 종료 후에는 운영 화면에서 사라지고 히스토리로 이동한다.

### 23.7 모바일/좁은 화면 기준이 필요함

선생님은 PC보다 모바일로 확인할 가능성이 있다.

추가 원칙:

- 선생님 화면은 모바일 우선으로 읽기 쉽게 구성한다.
- 관리자 화면은 PC 중심이지만 태블릿에서도 주요 확인이 가능해야 한다.
- 큰 표는 모바일에서 카드형으로 전환한다.
- 다운로드 버튼은 문서별로 명확하게 분리한다.

### 23.8 빈 상태와 예외 상태 화면이 필요함

아직 신청하지 않은 사용자, 선정되지 않은 사용자, 종료된 프로젝트만 있는 사용자도 있다.

필요한 빈 상태:

- 신청 가능한 행사가 없음
- 아직 신청한 꿈프가 없음
- 신청 접수 후 검토 중
- 미선정
- 출품 확인 전
- 활동확인서 발급 전
- 심사표 반영 전
- 종료된 프로젝트 없음

UX:

- 빈 화면에는 다음 행동을 하나만 보여준다.
- “데이터 없음” 대신 상황을 설명하고 다음 버튼을 제공한다.

### 23.9 화면 상태 체계가 필요함

각 페이지마다 상태가 제각각이면 사용자가 헷갈린다.

공통 상태 표현:

- 대기
- 진행 중
- 확인 필요
- 완료
- 보류
- 미대상
- 실패

UX:

- 같은 상태는 모든 화면에서 같은 색과 문구를 사용한다.
- 선생님 화면에서는 `실패`보다 `확인 필요` 표현을 우선한다.

## 25. 우선 개발 순서 제안

모든 화면을 한 번에 만들기보다 실제 운영 흐름에 맞춰 나눈다.

1차 개발:

- 관리자 꿈프 대시보드
- 관리자 행사 운영
- 선생님 신청 가능한 행사
- 선생님 신청서 작성
- 신청/선정
- 선생님 내 대시보드

2차 개발:

- 출품 확인
- 학생/작품 관리
- 메일/공지
- 혜택/지원
- 문의/요청

3차 개발:

- 활동확인서
- 확인서 템플릿 관리
- 심사 결과
- 심사표
- 히스토리
- 종료 전 점검

4차 개발:

- 권한 관리
- 작업 이력
- 쿠폰 재고 고도화
- 연동 로그 고도화
- 모바일 최적화
