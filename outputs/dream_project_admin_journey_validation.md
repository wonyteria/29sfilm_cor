# 29 WITH - 꿈프 관리자 활동 동선 및 기능 연계 검토

## 1. 검토 목적

꿈프 관리자가 29 WITH에서 행사를 생성하고, 모집을 시작하고, 신청자를 선정하고, 출품을 확인하고, 활동확인서/심사표/혜택 지급/종료까지 처리하는 흐름이 끊기지 않는지 확인한다.

검토 기준:

- 관리자가 오늘 처리해야 할 일을 바로 알 수 있는가?
- 한 단계의 처리 결과가 다음 단계에 자동으로 연결되는가?
- 선생님 화면에 보여져야 할 정보가 정확히 반영되는가?
- 외부 29초영화제/29역숏폼왕 관리자 페이지 연동이 행사별로 안전하게 작동하는가?
- 오류 발생 시 관리자가 빠르게 조치할 수 있는가?
- 종료 후 히스토리와 학교별 참여 이력이 정확히 남는가?

---

## 2. 관리자 전체 활동 동선

```text
로그인
→ 전체 대시보드 확인
→ 꿈프 대시보드 진입
→ 행사 생성/연결
→ 외부 연동 설정
→ 메일 템플릿/예약 메일 확인
→ 꿈프 모집 시작
→ 신청자 확인
→ 선정/예비/미선정 처리
→ 쿠폰 지급 및 담당자 페이지 생성
→ 출품 진행 중 자동 동기화 확인
→ 미매칭/오류 건 조치
→ 출품 마감 후 최종 출품 리스트 승인
→ 활동확인서 PDF 생성 및 발송
→ 수상작 발표 후 심사 결과 동기화
→ 심사표 PDF 생성 및 발송
→ 간식비 지급 예정일/상태 입력
→ 문의/요청 처리
→ 종료 전 점검
→ 꿈프 종료
→ 히스토리 저장 및 학교 참여 이력 반영
```

관리자 동선은 8단계로 나눈다.

1. 행사 준비
2. 모집 시작
3. 신청/선정
4. 출품 진행 관리
5. 마감 후 최종 승인
6. 문서/심사 결과 발급
7. 혜택/문의 처리
8. 종료/히스토리

---

## 3. 1단계: 행사 준비

### 3.1 관리자 행동

```text
꿈프 대시보드 진입
→ 행사 생성 또는 기존 행사 연결
→ 행사 정보 입력
→ 모집 목표 작품 수 입력
→ 외부 관리자 페이지 URL 입력
→ 서버 접근 정보 등록
→ 연동 테스트
```

### 3.2 필요한 화면

- 꿈프 대시보드
- 행사 운영
- 연동 설정
- 확인서 템플릿 관리

### 3.3 입력/연결 데이터

- 행사명
- 행사 유형
- 포스터
- 공모기간
- 출품 마감일
- 주제
- 홈페이지 URL
- 출품 URL
- 안내사항
- 모집 목표 작품 수
- 외부 관리자 페이지 URL
- 외부 행사 식별값
- 서버 접근 정보
- 필드 매핑 정보

### 3.4 오류 가능 지점

#### 문제 1. 여러 29초영화제/29역숏폼왕이 동시에 진행됨

위험:

- 학교명만으로 출품작을 가져오면 다른 행사 출품작이 섞일 수 있다.

대응:

- 외부 연동 설정에서 행사 식별값을 필수로 저장한다.
- 매칭 기준은 반드시 아래 구조여야 한다.

```text
external_event_id + dream_event_id + school_affiliation_name
```

관리자 화면 체크:

- 연동 설정 화면에 `외부 행사 식별값 확인 완료` 상태 표시
- 행사 식별값이 없으면 자동 동기화 버튼 비활성화

#### 문제 2. 외부 관리자 페이지 URL만 입력하고 서버 접근 정보가 없음

위험:

- 화면 URL만으로는 안정적인 데이터 수집이 어려움

대응:

- URL 등록과 서버 접근 정보 등록을 분리한다.
- 서버 정보가 없으면 상태를 `연동 준비 중`으로 표시한다.
- 관리자에게 “URL 등록 완료, 서버 접근 정보 필요”를 보여준다.

#### 문제 3. 29역숏폼왕에 팀명 필드가 아직 없음

위험:

- 팀명 기준 자동 매칭 불가

대응:

- 연동 설정에서 필수 필드 매핑 상태를 보여준다.
- `팀명 필드 없음`이면 자동 매칭을 비활성화하고 `필드 추가 필요`로 표시한다.

### 3.5 검토 결과

행사 준비 단계에서 가장 중요한 것은 `행사 식별값`과 `필드 매핑`이다.

이 둘이 확인되지 않으면 모집은 시작할 수 있어도, 자동 출품 확인은 시작하지 못하게 해야 한다.

## 4. 2단계: 모집 시작

### 4.1 관리자 행동

```text
행사 정보 최종 확인
→ 꿈프 모집 시작 버튼 클릭
→ 신청 버튼 활성화
→ 꿈프 회원 대상 안내 메일 발송
→ 행사 상태 모집중 변경
```

### 4.2 버튼 동작

`꿈프 모집 시작` 버튼은 단순 상태 변경이 아니다.

실행되어야 하는 작업:

```text
신청 버튼 활성화
행사 상태 예정 → 모집중
모집 시작일 기록
꿈프 회원 대상 안내 메일 발송
예약 메일 또는 즉시 발송 대상 생성
메일 발송 이력 저장
관리자 작업 이력 저장
```

### 4.3 오류 가능 지점

#### 문제 1. 모집 시작 전 필수 정보 누락

필수 확인:

- 행사명
- 행사 유형
- 공모기간
- 출품 마감일
- 홈페이지 URL
- 출품 URL
- 모집 목표 작품 수

대응:

- 누락 항목이 있으면 `꿈프 모집 시작` 버튼 비활성화
- 버튼 근처에 누락 항목 표시

#### 문제 2. 메일 발송 실패

대응:

- 모집 시작 자체는 완료 처리하되, 메일 실패 건은 관리자 작업함에 남긴다.
- 실패 대상만 재발송 가능해야 한다.

#### 문제 3. 발송 전 메일 내용, 날짜, 수신자를 확인하지 못함

위험:

- 잘못된 날짜에 메일이 발송됨
- 잘못된 수신자에게 메일이 발송됨
- 행사명, 출품 URL, 출품 소속명 안내 문구가 잘못 발송됨

대응:

- 자동 메일은 `예약 메일`로 생성하고 발송 전 확인 가능해야 한다.
- 관리자는 메일 제목, 본문, 발송 예정일, 수신자 이메일을 수정할 수 있어야 한다.
- 모집 시작, 선정 처리, D-day 안내, 활동확인서 발급, 심사표 안내, 간식비 안내 모두 발송 전 미리보기를 제공한다.
- 발송 완료 후에는 원본 수정이 아니라 재발송으로 처리한다.

#### 문제 4. 실수로 모집 시작 클릭

대응:

- 확인 팝업에서 실행 결과를 보여준다.

팝업:

```text
꿈프 모집을 시작하시겠습니까?

실행 시 신청 버튼이 활성화되고 꿈프 회원에게 안내 메일이 발송됩니다.
```

버튼:

- `모집 시작`
- `취소`

### 4.4 검토 결과

모집 시작은 여러 자동 작업을 포함하므로 반드시 트랜잭션처럼 묶어야 한다.

메일 발송 실패는 전체 실패로 처리하지 말고 재발송 작업으로 분리하는 것이 좋다.

자동 메일은 운영자가 발송 전 확인할 수 있도록 `예약 메일`로 관리하는 것이 안전하다.

## 5. 3단계: 신청/선정

### 5.1 관리자 행동

```text
신청자 리스트 확인
→ 참여 이력/관리자 메모 확인
→ 선정/예비/미선정 처리
→ 선정/예비/미선정 메일 내용과 수신자 확인
→ 선정 안내 메일 발송
→ 구독권 쿠폰 지급
→ 선생님 담당자 페이지 생성
```

### 5.2 필요한 화면

- 신청/선정
- 회원/참여 학교
- 혜택/지원
- 메일/공지
- 문의/요청

### 5.3 선정 버튼 동작

`선정 처리` 실행 시 자동 처리:

```text
신청 상태 선정으로 변경
선정일 기록
선정 안내 메일 발송
미사용 구독권 쿠폰 자동 지급
쿠폰 상태 지급완료로 변경
선생님 담당자 페이지 생성
신청 작품 수만큼 작품 슬롯 생성
선생님 대시보드에 선정 완료 표시
공지/메일함에 선정 메일 저장
관리자 작업 이력 저장
```

메일 관련 처리:

```text
선정 처리 전 메일 미리보기
수신자 이메일 확인
필요 시 제목/본문/수신자 수정
즉시 발송 또는 예약 발송 선택
발송 이력 저장
```

### 5.4 오류 가능 지점

#### 문제 1. 쿠폰 부족

위험:

- 선정은 되었는데 구독권이 지급되지 않음

대응:

- 선정 처리 전 쿠폰 재고 확인
- 쿠폰 부족 시 경고 팝업
- 관리자가 `쿠폰 없이 선정 계속`을 선택할 수 있는지 정책 결정 필요

권장 정책:

- 선정 처리는 가능
- 구독권 상태는 `지급 예정`으로 표시
- 관리자 작업함에 `쿠폰 지급 필요` 생성

#### 문제 2. 같은 학교 중복 신청

위험:

- 같은 학교가 여러 담당자로 중복 신청

대응:

- 신청/선정 리스트에 `중복 가능성` 표시
- 학교명, 연락처, 출품 소속명 기준으로 중복 후보를 보여준다.
- 자동 병합하지 않고 관리자 확인 후 처리한다.

#### 문제 3. 선정 후 작품 슬롯 생성 실패

위험:

- 선생님 화면에 학생/작품 관리가 비어 있음

대응:

- 선정 처리 후 후속 생성 결과를 확인한다.
- 실패 시 관리자 작업함에 `담당자 페이지 생성 실패` 또는 `작품 슬롯 생성 실패`로 남긴다.

#### 문제 4. 예비/미선정 메일 누락

대응:

- 예비/미선정 처리도 메일 발송과 공지/메일함 기록을 포함한다.
- 메일 실패 시 재발송 가능해야 한다.

#### 문제 5. 선정 메일 수신자 또는 내용 오류

대응:

- 선정 처리 확인 팝업에서 발송 대상 수와 대표 수신자를 보여준다.
- `메일 미리보기`에서 행사명, 출품 URL, 출품 소속명/팀명, 쿠폰번호 변수를 확인한다.
- 발송 전 수신자 수정이 가능해야 한다.
- 발송 후 수정은 불가하며, 수정본 재발송으로 처리한다.

### 5.5 검토 결과

선정 처리는 꿈프 운영에서 가장 중요한 자동화 버튼이다.

단순 상태 변경으로 만들면 오류가 많이 생긴다.

## 6. 4단계: 출품 진행 관리

### 6.1 관리자 행동

```text
출품 확인 화면 확인
→ 전체 동기화 또는 자동 동기화 결과 확인
→ 학교별 미확인 작품 확인
→ 유사 이름 후보 검토
→ 수동 매칭 또는 선생님 확인 요청
→ D-day 안내 메일 발송 상태 확인
→ 예약된 D-day 메일 내용/발송일/수신자 확인
```

### 6.2 필요한 화면

- 출품 확인
- 연동 설정
- 메일/공지
- 문의/요청

### 6.3 자동 동기화 기준

29초영화제:

```text
외부 행사 식별값 + 소속명 + 작품명 + 감독명
```

29역숏폼왕:

```text
외부 행사 식별값 + 팀명 + 작품명 + 출품자명
```

꿈프 매칭 기준:

```text
dream_event_id + external_event_id + 출품 소속명/팀명 완전 일치
```

### 6.4 오류 가능 지점

#### 문제 1. 외부 동기화 실패

대응:

- 관리자 화면에 실패 원인 표시
- `다시 동기화` 버튼 제공
- 선생님 화면에는 `관리자 확인 중`으로 표시

#### 문제 2. 유사 이름 후보를 자동으로 매칭

위험:

- 잘못된 학교에 작품이 연결될 수 있음

대응:

- 완전 일치만 자동 매칭
- 유사 이름은 관리자 후보 목록에만 표시
- 수동 매칭 시 이력 저장

#### 문제 3. 신청 작품 수보다 많은 작품 확인

대응:

- 관리자 화면에 `초과 출품` 표시
- 활동확인서 포함 여부는 마감 후 최종 승인에서 결정

#### 문제 4. 신청 작품 수보다 적은 작품 확인

대응:

- 선생님에게 D-day 안내
- 관리자 화면에 `미확인 작품 있음` 표시
- 마감 후 미이행 여부 판단에 반영

#### 문제 5. D-day 메일 중복 발송

대응:

- D-14, D-10, D-5, D-1 메일은 행사+학교+메일유형 기준으로 중복 방지
- 수동 재발송은 별도 이력으로 저장

#### 문제 6. D-day 메일 날짜가 잘못 잡힘

대응:

- D-day 메일은 출품 마감일 기준으로 자동 계산한다.
- 예약 메일 화면에서 실제 발송 예정일을 보여준다.
- 관리자는 필요한 경우 발송일을 수정할 수 있다.
- 출품 마감일이 변경되면 기존 예약 메일을 재계산할지 관리자에게 확인한다.

### 6.5 검토 결과

출품 진행 관리의 핵심은 `자동 매칭을 보수적으로 하고, 수동 조치를 쉽게 하는 것`이다.

---

## 7. 5단계: 출품 마감 후 최종 승인

### 7.1 관리자 행동

```text
출품 마감 확인
→ 학교별 최종 출품작 리스트 검토
→ 초과/미달/미확인 작품 확인
→ 학교별 최종 승인
→ 활동확인서 생성 가능 상태로 변경
```

### 7.2 승인 버튼 동작

`최종 승인` 실행 시:

```text
학교별 최종 작품 리스트 확정
활동확인서 발급 가능 상태 변경
선생님 화면에 출품 확인 완료 표시
발급 안내 메일 발송 또는 예약
작업 이력 저장
```

메일 관련 처리:

- 발급 안내 메일은 PDF 생성 후 자동 예약 또는 즉시 발송한다.
- 발송 전 수신자, 제목, 본문, 다운로드 링크를 미리보기로 확인할 수 있어야 한다.
- PDF 링크가 없으면 발송 버튼을 비활성화한다.

### 7.3 오류 가능 지점

#### 문제 1. 출품 마감 전 승인

대응:

- 출품 마감 전에는 `최종 승인` 비활성화
- 예외 승인 필요 시 권한 있는 관리자만 가능

#### 문제 2. 학생 이름 누락

대응:

- 활동확인서 생성 전 이름 누락 항목을 반드시 표시
- 누락이 있으면 PDF 생성 비활성화
- 선생님에게 이름 확인 요청 가능

#### 문제 3. 승인 후 리스트 변경

대응:

- 승인 후 변경은 `승인 취소` 또는 `재승인` 절차 필요
- 변경 이력 저장
- PDF가 이미 발급된 경우 재발급 요청으로 연결

### 7.4 검토 결과

최종 승인은 활동확인서 발급의 기준점이므로, 승인 전 검수 화면이 반드시 필요하다.

---

## 8. 6단계: 문서/심사 결과 발급

### 8.1 관리자 행동

```text
활동확인서 PDF 미리보기
→ PDF 생성
→ 발급 메일 발송
→ 수상작 발표 후 심사 결과 동기화
→ 순위/본심 여부 반영
→ 심사표 PDF 생성
→ 결과 안내 메일 발송
```

### 8.2 필요한 화면

- 활동확인서
- 확인서 템플릿 관리
- 심사 결과
- 메일/공지

### 8.3 오류 가능 지점

#### 문제 1. 템플릿이 없거나 깨짐

대응:

- 확인서 템플릿이 없으면 PDF 생성 버튼 비활성화
- 샘플 PDF 생성으로 템플릿 확인

#### 문제 2. PDF 생성 후 메일 발송 실패

대응:

- PDF 생성은 완료로 두고 메일만 재발송 가능하게 한다.
- 선생님 화면에는 PDF 다운로드 가능 상태를 유지한다.

#### 문제 5. PDF 링크가 빠진 메일 발송

대응:

- 활동확인서/심사표 안내 메일은 PDF 링크 검증 후 발송 가능하게 한다.
- 링크가 없으면 `메일 발송` 버튼을 비활성화한다.
- 메일 미리보기에서 다운로드 링크가 실제로 표시되는지 확인한다.

#### 문제 3. 심사 결과를 잘못된 행사에서 가져옴

대응:

- 심사 결과 동기화도 행사 식별값 필수
- 29초영화제와 29역숏폼왕 필드 매핑을 분리한다.

#### 문제 4. 29역숏폼왕에 본심 열 표시

대응:

- 행사 유형이 29역숏폼왕이면 본심 필드를 숨긴다.

### 8.4 검토 결과

문서 발급은 관리자에게 미리보기와 재발급 흐름이 꼭 필요하다.

---

## 9. 7단계: 혜택/문의 처리

### 9.1 관리자 행동

```text
쿠폰 재고 확인
→ 미지급 학교 확인
→ 간식비 지급 예정일 입력
→ 간식비 안내 메일 내용/발송일/수신자 확인
→ 보류/미대상 사유 입력
→ 선생님 문의 처리
```

### 9.2 필요한 화면

- 혜택/지원
- 쿠폰 재고 관리
- 문의/요청
- 회원/참여 학교

### 9.3 오류 가능 지점

#### 문제 1. 간식비 예정일 미입력

대응:

- 꿈프 대시보드 오늘 할 일에 `간식비 지급 예정일 미입력` 표시
- 미입력 학교 필터 제공

#### 문제 2. 보류 사유가 선생님에게 너무 내부적으로 노출

대응:

- 관리자 내부 사유와 선생님 노출 문구를 분리한다.

예시:

```text
관리자 내부 사유: 통장사본 식별 불가
선생님 노출 문구: 간식비 지급을 위해 증빙 확인이 필요합니다.
```

#### 문제 3. 문의가 메일/전화/시스템으로 흩어짐

대응:

- 선생님 화면의 주요 문제는 `문의/요청`으로 구조화한다.
- 관리자는 요청 유형별로 처리한다.

### 9.4 검토 결과

혜택과 문의는 운영 후반에 몰리므로, 대시보드에서 미처리 건을 강하게 보여줘야 한다.

---

## 10. 8단계: 종료/히스토리

### 10.1 관리자 행동

```text
종료 전 점검
→ 미완료 항목 확인
→ 예외 처리
→ 꿈프 종료
→ 히스토리 저장
→ 학교별 참여 이력 업데이트
```

### 10.2 종료 전 체크리스트

종료 전 필수 확인:

- 신청/선정 상태 확정
- 최종 출품 리스트 승인
- 활동확인서 발급 완료 또는 미대상 처리
- 심사표 발급 완료 또는 미대상 처리
- 구독권 지급 상태 확인
- 간식비 지급 상태 입력
- 미이행 학교 반영
- 문의/요청 미처리 없음
- 메일 발송 실패 건 처리

### 10.3 종료 버튼 동작

`꿈프 종료` 실행 시:

```text
꿈프 상태 종료로 변경
운영 화면에서 진행 중 목록 제거
히스토리 저장
학교별 참여 이력 업데이트
선생님 참여 프로젝트 종료 영역으로 이동
문서 다운로드 유지
관리자 작업 이력 저장
```

### 10.4 오류 가능 지점

#### 문제 1. 미완료 항목이 있는데 종료

대응:

- 종료 전 점검 화면에서 미완료 항목 표시
- 필수 항목 미완료 시 종료 버튼 비활성화

#### 문제 2. 종료 후 선생님이 문서 다운로드 불가

대응:

- 종료 후에도 선생님 종료 프로젝트에서 PDF 다운로드 유지

#### 문제 3. 참여 이력 업데이트 누락

대응:

- 종료 처리에 참여 이력 업데이트를 포함
- 실패 시 종료 완료가 아니라 `종료 처리 중 오류`로 표시

### 10.5 검토 결과

종료는 단순 보관이 아니라 다음 꿈프 선정 기준에 영향을 주는 중요한 처리다.

## 11. 기능 간 연계 체크리스트

### 11.1 행사 준비 → 모집 시작

- 행사 필수 정보가 모두 입력되었는가?
- 모집 목표 작품 수가 입력되었는가?
- 출품 마감일이 입력되었는가?
- 외부 연동 정보 상태가 표시되는가?
- 모집 시작 시 신청 버튼이 활성화되는가?
- 모집 안내 메일이 발송 또는 실패 이력으로 남는가?
- 모집 안내 메일의 제목, 본문, 발송일, 수신자를 발송 전 확인/수정할 수 있는가?

### 11.2 모집 시작 → 신청/선정

- 선생님이 신청 가능한 행사에서 해당 행사를 볼 수 있는가?
- 신청 완료 후 관리자 신청/선정 리스트에 표시되는가?
- 신청 완료 후 선생님 대시보드에 프로젝트 카드가 생성되는가?
- 목표 작품 수 초과 안내가 표시되는가?

### 11.3 선정 → 선생님 담당자 페이지

- 선정 상태가 변경되는가?
- 선정 메일이 발송되는가?
- 선정 메일의 수신자와 본문을 발송 전 확인할 수 있는가?
- 구독권 쿠폰이 지급 또는 지급 예정으로 표시되는가?
- 작품 슬롯이 생성되는가?
- 선생님 대시보드에 선정 완료가 표시되는가?

### 11.4 출품 동기화 → 출품 확인

- 행사 식별값 기준으로 해당 행사 출품작만 가져오는가?
- 29초영화제는 소속명 기준인가?
- 29역숏폼왕은 팀명 기준인가?
- 완전 일치만 자동 매칭되는가?
- 유사 후보는 수동 매칭 대상으로만 표시되는가?
- 수동 매칭 이력이 저장되는가?

### 11.5 최종 승인 → 활동확인서

- 출품 마감 후에만 최종 승인 가능한가?
- 학생 이름 누락 시 PDF 생성이 막히는가?
- 승인 후 선생님 화면에 다운로드 가능 상태가 반영되는가?
- 발급 안내 메일이 발송되는가?

### 11.6 심사 결과 → 심사표

- 수상작 발표 후 결과 동기화가 가능한가?
- 행사 유형별 필드 매핑이 분리되는가?
- 29초영화제는 본심 여부가 표시되는가?
- 29역숏폼왕은 본심 여부가 숨겨지는가?
- 결과 안내 메일이 발송되는가?
- 활동확인서/심사표 안내 메일에 다운로드 링크가 포함되었는가?

### 11.7 혜택/지원 → 선생님 대시보드

- 쿠폰 지급 상태가 반영되는가?
- 간식비 예정일이 대시보드에 표시되는가?
- 보류/미대상 안내가 선생님용 문구로 표시되는가?
- 지급 예정일 변경 이력이 남는가?
- 간식비 안내 메일 발송일과 수신자를 수정할 수 있는가?

### 11.8 종료 → 히스토리

- 종료 전 체크리스트가 있는가?
- 미완료 항목이 있으면 종료가 막히는가?
- 종료 후 진행 중 목록에서 빠지는가?
- 히스토리에 저장되는가?
- 선생님 종료 프로젝트에 문서 다운로드가 남는가?
- 학교별 참여 이력이 업데이트되는가?

## 12. 예상되는 주요 오류와 방지 방법

| 오류 | 원인 | 방지 방법 |
|---|---|---|
| 다른 행사 출품작이 섞임 | 학교명만으로 매칭 | 행사 식별값 + 소속명/팀명 기준 |
| 모집 시작했는데 신청 버튼 비활성 | 모집 상태 변경 누락 | 모집 시작 버튼에 신청 활성화 포함 |
| 선정됐는데 담당자 페이지 없음 | 선정 후 생성 작업 누락 | 선정 처리에 페이지/슬롯 생성 포함 |
| 쿠폰 지급 실패 | 쿠폰 부족 | 지급 예정 상태와 관리자 작업함 생성 |
| D-day 메일 중복 발송 | 중복 방지 키 없음 | 행사+학교+메일유형 기준 중복 방지 |
| 유사 이름 자동 매칭 오류 | 과도한 자동화 | 완전 일치만 자동, 유사 후보는 수동 |
| 확인서에 학생 이름 누락 | 외부 데이터 누락 | PDF 생성 전 필수값 검증 |
| 심사표에 본심 열 오류 | 행사 유형 미반영 | 유형별 화면/필드 분기 |
| 종료 후 이력 미반영 | 종료와 이력 업데이트 분리 | 종료 버튼에 이력 업데이트 포함 |

### 13.3 작업 이력이 필요한 행동

- 모집 시작

- 쿠폰 지급
- 학교명 변경 승인
- 출품 소속명/팀명 변경 승인
- 수동 매칭
- 최종 승인
- PDF 생성/재발급
- 메일 재발송
- 간식비 상태 변경
- 꿈프 종료

## 14. 최종 판단

관리자 동선은 전체적으로 자연스럽게 구성 가능하다.

다만 안정적인 운영을 위해 아래 6개 연결은 반드시 하나의 흐름으로 묶어야 한다.

1. 모집 시작 → 신청 활성화 + 안내 메일 + 상태 변경
2. 선정 처리 → 상태 변경 + 메일 + 쿠폰 + 담당자 페이지 + 작품 슬롯
3. 출품 동기화 → 행사 식별값 + 소속명/팀명 완전 일치 매칭
4. 최종 승인 → 출품 리스트 확정 + 확인서 발급 가능 상태
5. 심사 결과 반영 → 순위/본심 표시 + PDF + 안내 메일
6. 꿈프 종료 → 히스토리 저장 + 학교별 참여 이력 업데이트

관리자 화면의 핵심은 “많은 기능을 한 화면에 보여주는 것”이 아니라, 현재 단계에서 해야 할 작업을 놓치지 않게 하는 것이다.

따라서 모든 관리자 화면은 아래 구조를 유지해야 한다.

```text
현재 행사 상태
처리 필요 건수
다음에 눌러야 할 버튼
오류/예외 조치
작업 이력
```
